# VoxMate AI — Правки: экономика обработки, вход, обновление PWA

**Дата:** 13 июля 2026
**Версия:** 1.0
**Назначение:** закрыть три пробела, выявленных при ревью бэклога: (1) отсутствие модели тарифов/лимитов/учёта расхода, (2) неоднозначность первичного идентификатора при входе через Telegram/ВКонтакте, (3) отсутствие стратегии обновления и кэширования PWA.

> **Как читать:** ниже — три новых ADR (033–035), вставки в NFR, уточнение ADR-027, дополнения в глоссарий, риски и порядок реализации. Все три пункта — осознанные решения, требующие подтверждения владельцем продукта только там, где явно помечено ⚠️.

## 2. ADR-033 — Экономика обработки: квоты и учёт расхода

### 2.1. Проблема
После регистрации пользователь получает безлимитные 4-часовые встречи с облачным хранением до года. Каждая минута ASR + LLM через OpenRouter — прямые денежные затраты. Без квот и учёта один-два пользователя способны исчерпать бюджет пилота, а экономику продукта нельзя спрогнозировать.

### 2.2. Единица учёта
**Минута обработки (audio-minute)** — и ASR, и LLM масштабируются от длительности. Считается по **фактической распознанной длительности**, а не по размеру файла.

### 2.3. Учёт расхода — `usage_ledger`
Каждая обработка пишет запись расхода:
- `meeting_id`, `account_id`;
- `asr_minutes`;
- `llm_tokens_in`, `llm_tokens_out`;
- стоимость по прайсу провайдера **на момент операции** (провайдер и цена фиксируются, чтобы задним числом расход не «плыл»);
- `provider`, `model`, `created_at`.

Даёт основу для будущего прайсинга, для алертов и для метрики себестоимости.

### 2.4. Квоты уровня аккаунта (пилот)
- месячный лимит минут обработки;
- лимит параллельных задач в очереди;
- дневной предохранитель (rate-cap).

Поведение при достижении:
- **софт-лимит** → предупреждение, обработка продолжается;
- **хард-лимит** → постановка новой обработки блокируется; **запись при этом не теряется** — она уже сохранена в S3 (offline-first, этап 1) и переходит в статус **«ожидает квоты»**.

> Важно совместить с offline-first: запись всегда сохраняется, гейтится **только обработка**, а не запись.

### 2.5. Предохранители на стороне воркера
- **cost-ceiling на одну встречу**: 4-часовая встреча с map-reduce не должна превышать порог по токенам. При превышении — деградация: более дешёвая модель / грубее чанкинг / стоп с явным статусом (не молчаливое сжигание бюджета);
- **глобальный дневной budget-cap** на весь сервис с алертом в мониторинг.

### 2.6. Демо
Остаётся без изменений: ~1 мин, без облачного хранения (ADR-009).

### 2.7. Новые события и метрики
- события: `quota_soft_warning`, `quota_hard_block`, `processing_cost_exceeded`, `processing_waiting_quota`;
- продуктовые метрики: `cost_per_meeting_minute`, `cost_per_activated_user`, `quota_hit_rate`.

Аналитика по-прежнему **не содержит** аудио/текста/имён/контактов (152-ФЗ).

### 2.8. ⚠️ Требует решения владельца продукта
Пилот идёт с **фиксированными квотами на всех** или уже с **тарифными планами**?
От этого зависит, нужен ли платёжный провайдер в MVP.
**Рекомендация:** для пилота — фиксированные квоты без платежей. Платёжный контур — отдельный крупный объём, выносится за MVP.

## 3. ADR-034 — Email как обязательный первичный идентификатор

### 3.1. Проблема
ADR-027 («все шесть провайдеров входа») не различает **первичную регистрацию** и **привязку**. Telegram OAuth email не отдаёт в принципе; ВКонтакте — не гарантированно (зависит от прав и настроек аккаунта). Без стабильного идентификатора ломается восстановление доступа (в решениях оно завязано на email-ссылку) и объединение аккаунтов.

### 3.2. Правило
**Первичная регистрация — только через провайдеров с верифицированным email:**
Email/пароль, Яндекс ID, Google, Сбер ID.

**Telegram и ВКонтакте — только привязка** к уже существующему аккаунту, не самостоятельная регистрация.

### 3.3. Сценарий «первый вход через Telegram/ВК»
Если email недоступен — **не создавать пустой аккаунт**, а вести пользователя на шаг **«подтвердите email»**: он вводит email, подтверждает его, и уже к этому аккаунту привязывается Telegram/ВК как способ входа. Так верифицированный email есть всегда.

### 3.4. Модель данных
- `email` — обязательный уникальный атрибут аккаунта;
- `auth_identities` (`provider`, `provider_user_id`, `account_id`) — many-to-one к аккаунту;
- при коллизии email — предложить **привязку** к существующему аккаунту, а не создавать дубль.

### 3.5. Связь с уже принятыми решениями
Уточняет (не отменяет) ADR-027 — все шесть провайдеров остаются, но с разделением ролей «регистрация / привязка». Стыкуется с решением «восстановление доступа: email-ссылка + вход через привязанный OAuth».

## 4. ADR-035 — Обновление и кэширование PWA

### 4.1. Проблема
Агрессивный Service Worker с cache-first закэширует старую сборку — пользователь застрянет на устаревшей версии, с риском рассинхрона API-контракта (клиент старый, бэкенд новый). Особенно опасно на фоне offline-first (этап 1), где кэш нужен намеренно.

### 4.2. Стратегии кэша по типам ресурсов
| Ресурс | Стратегия | Причина |
|--------|-----------|---------|
| HTML-оболочка, навигация | network-first / stale-while-revalidate с коротким таймаутом | новая версия подхватывается при первом онлайне |
| Статические ассеты (JS/CSS) | cache-first, **только с content-hash в именах файлов** | новый билд = новые URL, старьё не липнет |
| API | сеть, без кэширования ответов | офлайн-запись живёт в IndexedDB отдельно от SW-кэша |

### 4.3. Механика обновления Service Worker
- новый SW встаёт в `waiting`;
- приложение показывает **ненавязчивый баннер «Доступна новая версия — обновить»**;
- по нажатию — `skipWaiting()` + `clients.claim()` + перезагрузка;
- **во время активной записи обновление откладывается** до её завершения (не прерывать сценарий молчаливой перезагрузкой).

### 4.4. Защита от рассинхрона контракта
- клиент шлёт **версию сборки в заголовке** запроса;
- бэкенд при несовместимой версии отвечает специальным кодом → приложение показывает «обновите приложение» и триггерит обновление SW;
- старые версии API держатся совместимыми на время выката.

### 4.5. Kill-switch (обязательно)
Если выкатили битый SW, нужен способ сбросить его:
- версионирование имени кэша `cache-v{N}` с очисткой старых на `activate`;
- в крайнем случае — серверный флаг, по которому клиент делает `unregister`.

### 4.6. Куда встроить
В **Этап 1** порядка реализации — offline-first и Service Worker собираются там же; иначе кэш-стратегию придётся переписывать задним числом.

## 5. Правки существующих файлов

### 5.1. Уточнение ADR-027 (в `VoxMate_Решения_риски_порядок.md`)
Формулировку дополнить: «все шесть провайдеров входа сохраняются; при этом регистрация — только через провайдеров с верифицированным email (Email/Яндекс/Google/Сбер ID), Telegram и ВКонтакте — только привязка (см. ADR-034)».

### 5.2. Вставка в NFR (`VoxMate_NFR_и_метрики.md`)

**Новый раздел «Экономика обработки»:**
- единица учёта — минута обработки;
- `usage_ledger` на каждую обработку;
- квоты аккаунта: месячный лимит минут, лимит параллельных задач, дневной rate-cap;
- предохранители воркера: cost-ceiling на встречу, дневной budget-cap;
- метрики: `cost_per_meeting_minute`, `cost_per_activated_user`, `quota_hit_rate`.

**Новый раздел «Версионирование PWA / клиент↔сервер»:**
- разделение стратегий SW-кэша по типам ресурсов;
- версия сборки в заголовке; несовместимость → принудительное обновление;
- kill-switch на битый SW.

### 5.3. Новые термины (в `VoxMate_Глоссарий.md`)
- **usage_ledger** — журнал расхода обработки (минуты ASR, токены LLM, стоимость по прайсу провайдера на момент операции). Основа учёта и алертов.
- **Квота обработки** — лимит минут/параллельных задач на аккаунт; при исчерпании блокируется обработка, но не запись.
- **auth_identities** — таблица связок «провайдер входа → аккаунт»; несколько способов входа на один аккаунт с общим email.
- **Service Worker** — фоновый скрипт PWA, управляющий кэшем и офлайн-работой; требует явной стратегии обновления, чтобы не залипала старая сборка.
- **Kill-switch (PWA)** — механизм принудительного сброса битого Service Worker (версия кэша + серверный флаг unregister).

## 6. Дополнение к рискам (`VoxMate_Решения_риски_порядок.md` §4)

| Риск | Уровень | Мера |
|------|---------|------|
| Неконтролируемый расход ASR+LLM (OpenRouter) | Высокий | Квоты аккаунта + `usage_ledger` + cost-ceiling на встречу + дневной budget-cap (ADR-033) |
| Вход через Telegram/ВК без email → потеря доступа | Высокий | Email — обязательный первичный ключ; TG/ВК только привязка (ADR-034) |
| Залипание старой сборки PWA / рассинхрон контракта | Средний | Стратегии SW-кэша + версия сборки в заголовке + kill-switch (ADR-035) |
