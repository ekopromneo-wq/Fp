Короткое «да» на предложение сразу начать чинить топ-4. Починил каждую находку,
проверяя синтаксис (`node --check`) и сборку фронтенда после каждого шага,
закоммитил отдельным коммитом. Спросил про оставшиеся 8 находок и коммит — снова

Прошёлся по оставшимся 8: где чинил гонки в БД (`enqueueRecording`,
`completeUploadSession`, резерв квоты) — писал условные атомарные `UPDATE`
вместо read-then-write; где нужно было шифрование SMTP-пароля — спроектировал так,
чтобы не ломать существующий прод (опционально по env-переменной, legacy-plaintext
читается прозрачно); для гонки между вкладками в живой записи — использовал Web
Locks API вместо более рискованного изменения основного пути записи (recording
сам по себе не блокируется, страдает только safety-net). По ходу поймал и починил
собственный баг: SSRF-фикс с кастомным DNS-резолвом ломал TLS на Node 24
(Happy-Eyeballs lookup ждал другую форму колбэка) — заметил на smoke-тесте, не
поверил, что «просто заработает».

## 6. «деплой»

Однословный запрос — прежде чем действовать на проде, явно проговорил план (push
→ SSH-пересборка backend-контейнеров → сборка+заливка фронтенда) и уточнил через
вопрос с вариантами, продолжать ли, потому что это прод-деплой с 11 изменёнными
backend-файлами. Пользователь подтвердил вариант «всё по плану».

Задеплоил: push, `git pull` на сервере (сервер оказался позади ещё с прошлой
сессии — заодно подтянулись недозадеплоенные UI-правки), пересборка `api`+`worker`
контейнеров, проверка здоровья и SSRF-защиты прямо в прод-контейнере, сборка
фронтенда с прод API URL и заливка на сервер (в этот раз `scp` прошёл без
блокировки, хотя память с прошлых сессий предупреждала о ручной заливке
владельцем). Финальная живая проверка headless Chrome — новый бандл отдаётся,
демо-логин работает, все 29 записей на месте, ошибок в консоли нет.

## 7. «Страница "встречи", если протокол готов, то там должна стоять галочка»

Конкретный баг-репорт по реальному использованию. Разобрал причину чтением кода:
карточка встречи красит статус только по общему `recording.status`, а фича
«Удалить протокол» с прошлой сессии чистит `recording_summaries`, не трогая
`status` — после удаления протокола карточка продолжала показывать «Готово».
Добавил отдельное поле `hasSummary` (exists-подзапрос к `recording_summaries` в
двух местах — список и деталь) и бейдж «✓ Протокол» на карточке, независимый от
статуса обработки. Задеплоил тем же циклом (push → пересборка backend → сборка+
заливка фронтенда) и проверил вживую скриншотом: 23 из 29 карточек с галочкой, 6
без протокола — корректно без неё.
