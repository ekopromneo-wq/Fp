## 2. «Давай по порядку» → Gate перед публичным запуском (§6)

Сверились со спекой: функциональный MVP закрыт, остался чек-лист перед релизом.
Пошли по нему.

- **§6.3 Удаление.** Каскад проверил транзакцией с `ROLLBACK` на прод-БД (ничего
  не сохранено): удаление аккаунта чистит все дочерние данные; живой FK —
  `ON DELETE CASCADE`. **Находка:** на проде не было бэкапов вообще. По твоему
  решению «настроить сейчас» — сделал `deploy/backup.sh` (pg_dump + tar тома
  MinIO, ротация 14 дн, cron 03:30), проверил целостность и **восстановление** в
  одноразовую БД (счётчики совпали).
- **§6.4 Обрыв сети.** Возобновляемую upload-сессию проверил вживую скриптом:
  частичная загрузка переживает «обрыв», докачка с офсета, guard неверного
  офсета (409), сборка файла. Клиентскую очередь сверил чтением.

## 4. Gate дальше: DOCX, безопасность, нагрузка

- **§6.8 DOCX.** Прогнал генератор протокола на 5 фикстурах (пустой → большой на
  93 тыс. символов), распаковал `word/document.xml`: XML валиден, кириллица цела,
  спецсимволы экранированы, маркер спорного решения на месте. Закрыт.
- **§6.7 Security review OAuth.** Ручной аудит нашёл 4 находки; по твоему «исправить
  все 4» — задеплоил: OAuth больше не обходит закрытую регистрацию; линковка к
  аккаунту только по подтверждённому email; internal-callback режется на Caddy +
  constant-time; Telegram-HMAC timing-safe. Всплыл **нюанс деплоя Caddy** (bind-
  mount одиночного файла пинит inode — нужен `--force-recreate`, не `reload`);
  записал в память.
- **§6.6 Нагрузка 1 ГБ.** Анализом нашёл **блокер памяти**: завершение чанковой
  загрузки держало весь файл в памяти дважды (`readFile` + `Buffer.from`) → на
  1 ГБ пик ~2 ГБ RAM → своп/OOM. По твоему «реализовать сейчас» — потоковый путь
  (ffprobe по staging-файлу + стрим в MinIO). Проверил вживую: 150 МБ → пик api
  160 MiB вместо ~370; память O(chunk). Полный платный 4ч-ASR-прогон оставили на
  решение о бюджете.
