# Приёмка качества извлечения задач (Gate §6.5)

Харнесс для замера **precision / recall / F1** извлечения задач против эталонной
разметки. Закрывает верификационную часть Gate §6.5 (`VoxMate_Решения_риски_порядок.md`)
и NFR §5. Сам замер требует **реального размеченного датасета** — код здесь готов,
данные собираются людьми (см. ниже).

## Цели (утверждены, ADR-031)

| Метрика | Цель | Почему |
|---------|------|--------|
| Precision | ≥ 0.90 | Приоритет: «хуже создать ложную задачу» |
| Recall | ≥ 0.85 | |
| F1 | ≥ 0.87 | Производная |

Это **критерий приёмки**, а не гарантия: до первого замера на реальном датасете
цифры не подтверждены.

## Датасет (ADR-032)

- **50 реальных встреч**, суммарно **≥ 40 часов**;
- **≥ 500 размеченных задач**;
- покрытие всех системных шаблонов протокола (`planning`, `negotiation`,
  `interview`, `project`, `meeting`, `freeform`);
- **2 разметчика независимо** + **арбитр** для спорных случаев → согласованный
  `gold.json`.

**Повторная приёмка обязательна** при смене: ASR-модели, LLM, промптов, логики
чанкинга/map-reduce, языка, шаблона протокола или логики разрешения сроков.

## Формат файлов

`gold.json` (эталон) и `pred.json` (предсказания системы) — одинаковой формы:

```json
{
  "meetings": [
    {
      "id": "m1",
      "title": "необязательно",
      "tasks": [
        { "assignee": "Иван", "description": "подготовить квартальный отчёт", "dueText": "до пятницы" }
      ]
    }
  ]
}
```

- Встречи сопоставляются по `id` (должны совпадать между файлами).
- В задаче значим `description` (обязателен) и `assignee` (если указан у обеих —
  влияет на сопоставление). `dueText`/`dueDate` пока не влияют на метрику
  (см. «Развитие»).
- Пример — в `fixtures/gold.example.json` и `fixtures/pred.example.json`.

## Как получить `pred.json`

Прогнать каждую из 50 встреч через Stenogram (тем же ASR/LLM/промптами, что в
проде) и выгрузить извлечённые задачи. Источник — таблица `recording_tasks`
(поля `assignee`, `description`, `due_text`) по каждой записи, сгруппировать по
встрече в формат выше. Экспорт-скрипт/эндпоинт пока не сделан — TODO.

## Запуск

```bash
node eval/score.mjs --gold gold.json --pred pred.json          # человекочитаемо
node eval/score.mjs --gold gold.json --pred pred.json --json   # машинный вывод
node eval/score.mjs --gold gold.json --pred pred.json --threshold 0.6
node eval/score.test.mjs                                        # self-test харнесса
```

Код выхода: `0` — все три цели достигнуты, `1` — нет.

## Правило сопоставления

Детерминированный базовый матчер (`score.mjs`):

1. **Похожесть описаний** — коэффициент Дайса по нормализованным токенам
   (нижний регистр, `ё→е`, без пунктуации, без коротких слов и стоп-слов).
2. **Защита по исполнителю** — если у обеих задач указан `assignee` и они разные,
   пара **не** считается совпадением (задача на другого человека — не та же
   задача; поддерживает приоритет precision).
3. **Жадное уникальное сопоставление** пар со скорингом ≥ `threshold` (по умолч. 0.5).

`TP` — сопоставленные пары, `FP` — лишние предсказания (в т.ч. задачи для встреч
вне эталона), `FN` — эталонные задачи без предсказания.

### Развитие (для арбитра)

Токенный матчер не ловит перефразы («подготовить отчёт» ≠ «сделать сводку»).
`pairScore` в `score.mjs` — единственная точка расширения: для точной приёмки
подключить **LLM-судью** (тот же OpenRouter, что в проде) — на вход пара
«эталон/предсказание», на выход «та же задача да/нет», с логированием для
арбитража спорных пар (ADR-032). Так же можно учесть совпадение срока (`dueText`).

## Что вне охвата

- **Качество транскрипции** (NFR §4, ≥90% текста) — отдельный замер WER, не здесь.
- Сбор и разметка датасета — ручная работа (2 разметчика + арбитр).
