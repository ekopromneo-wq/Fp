| Доступность | Drag-and-drop на ПК — есть; screen reader, foldable, haptic-настройка — откладываем. SSO не нужен. |
| Хранение по умолчанию | **Облако** (с возможностью переключить на «только устройство»). |
| Биометрия голоса | На паузе до юридической проверки 152-ФЗ. |
| Календарь | **Outlook / Microsoft 365** (Microsoft OAuth). |

**Приоритеты:** `MVP` / `След. релиз` / `Поздний этап` / `Вне продукта`.

## Роли (для MVP)

| Роль | Описание |
|------|----------|
| **Пользователь** | Владелец приложения. Записывает, обрабатывает, редактирует, отправляет. В персональной версии совмещает почти все роли. |
| **Исполнитель** | Человек, которому ставится задача. Может не иметь аккаунта — получает задачу через Telegram/Email/Б24. |
| **Внешний участник** | Получает только ссылку на результат (без регистрации). |

> Роль «Администратор организации» и командные роли появятся вместе с корпоративной версией (следующий этап).

---

## Общий Definition of Done (применяется к каждой истории)

1. Реализованы основной и ошибочные сценарии.
2. Данные не теряются при обновлении страницы и кратком обрыве сети.
3. Права проверяются на сервере (server-side ACL).
4. Есть состояния loading / empty / success / error.
5. Аналитика не содержит аудио, текста встреч, имён и контактов (152-ФЗ).
6. Проверено на Android 10+ (PWA) и Windows.
7. Основные действия доступны с клавиатуры; статусы не обозначаются только цветом.
8. Удаление требует подтверждения.
9. Для истории записаны критерии приёмки Дано/Когда/Тогда.
10. Соблюдён quality floor доступности (контраст WCAG AA, видимый focus, `prefers-reduced-motion`).

## Эпик 1. Запись аудио

### US-1.1 — Быстрый старт записи одной кнопкой
**Как** пользователь, **когда** начинается встреча, **я хочу** начать запись одним нажатием, **чтобы** не терять первые фразы на настройку.
**Приоритет:** MVP
- Триггер: тап по кнопке записи на главном экране (без подтверждения).
- Данные: источник (phone/computer), время старта, локальный ID.
- Во время записи видно: длительность + индикатор активности (красная кнопка + «столбики» уровня).
- Звуковой сигнал старта — по настройке пользователя.

**Критерии приёмки:**
- Дано главный экран, когда пользователь жмёт кнопку записи, тогда запись стартует сразу без диалога подтверждения.
- Дано идёт запись, когда пользователь смотрит на экран, тогда видны длительность и индикатор активной записи.
- Дано уровень звука ниже порога, тогда показывается предупреждение «звук слишком тихий».
- Дано включён экран записи, тогда экран не гаснет.

### US-1.2 — Пауза и продолжение
**Как** пользователь, **когда** встречу нужно прервать, **я хочу** поставить запись на паузу и продолжить, **чтобы** не создавать несколько файлов.
**Приоритет:** MVP
- Пауза/возобновление внутри одной записи.
- При входящем звонке — автопауза, затем продолжение.
- При блокировке телефона — запись продолжается.

**Критерии приёмки:**
- Дано идёт запись, когда приходит входящий звонок, тогда запись автоматически ставится на паузу и возобновляется после звонка.
- Дано телефон заблокирован, когда идёт запись, тогда запись продолжается.

### US-1.3 — Запись при свёрнутом приложении
**Как** пользователь, **когда** во время встречи нужно открыть другое приложение, **я хочу** чтобы запись не прерывалась, **чтобы** ничего не потерять.
**Приоритет:** MVP
- Критерий: Дано идёт запись, когда приложение свёрнуто, тогда запись продолжается.

### US-1.4 — Завершение записи
**Как** пользователь, **когда** встреча закончилась, **я хочу** остановить запись одной кнопкой и увидеть подтверждение сохранения, **чтобы** быть уверенным, что аудио не потеряется.
**Приоритет:** MVP
- Стоп — та же кнопка (была красная). Подтверждение завершения не требуется.
- После стопа — индикатор «аудио сохранено» (зелёная галочка).
- Завершение **необратимо**; название/проект сразу задавать не нужно (но можно).

**Критерии приёмки:**
- Дано идёт запись, когда пользователь жмёт стоп, тогда запись завершается и показывается зелёная галочка «сохранено».
- Дано запись завершена, тогда можно добавить к этой встрече ещё один фрагмент/файл.

### US-1.5 — Несколько фрагментов в одной встрече
**Как** пользователь, **когда** встреча шла в несколько заходов, **я хочу** добавить несколько записей/файлов в одну встречу, **чтобы** получить единый протокол.
**Приоритет:** MVP
- Объединение нескольких аудиофрагментов в одну встречу.

### US-1.6 — Предупреждения о проблемах со звуком
**Как** пользователь, **когда** микрофон закрыт или звук слишком тихий, **я хочу** получить предупреждение, **чтобы** не записать бесполезное аудио.
**Приоритет:** MVP
- Предупреждение при тихом/отсутствующем звуке; пользователь решает — продолжать или нет.

### US-1.7 — Выбор и тест микрофона (десктоп)
**Как** пользователь на компьютере, **когда** подключено несколько микрофонов, **я хочу** выбрать нужный и протестировать его, **чтобы** записать качественно.
**Приоритет:** MVP (выбор) / тест — опция в настройках.

### US-1.8 — Устойчивость к разряду устройства
**Как** пользователь, **когда** телефон разряжается во время записи, **я хочу** чтобы записанное сохранилось, **чтобы** не потерять встречу.
**Приоритет:** MVP
- Запись сохраняется на момент разрядки; периодическое автосохранение с интервалом из настроек.

**Критерии приёмки:**
- Дано идёт запись, когда устройство разряжается, тогда фрагмент до момента отключения сохранён и доступен.

### US-1.9 — Фото документов к встрече
**Как** пользователь, **когда** на встрече показывают документ, **я хочу** сфотографировать его и приложить к встрече, **чтобы** сохранить контекст.
**Приоритет:** След. релиз

## Эпик 2. Загрузка готового файла

### US-2.1 — Загрузка аудио/видео файла
**Как** пользователь, **когда** у меня уже есть запись, **я хочу** загрузить файл и получить протокол, **чтобы** не записывать заново.
**Приоритет:** MVP
- Форматы: MP3, M4A, WAV, OGG/OPUS, AAC, FLAC + видео MP4, MOV, MKV, WEBM (извлекаем звук).
- Максимум: размер 1 ГБ, длительность **4 часа**.
- Можно загрузить несколько файлов сразу и объединить в одну встречу.

**Критерии приёмки:**
- Дано выбран видеофайл, когда он загружен, тогда из него извлекается звук и запускается обработка.
- Дано загружены несколько файлов, тогда их можно объединить в одну встречу.

### US-2.2 — Обработка дублей и битых файлов
**Как** пользователь, **когда** файл дублируется или повреждён, **я хочу** понятное сообщение и выбор действия, **чтобы** не гадать, что пошло не так.
**Приоритет:** MVP
- Дубль → сообщение + выбор «сохранить оба / удалить дубль».
- Повреждённый файл → сообщение + предложение загрузить другой.
- Неподдерживаемый кодек → сообщение + предложение действия.

### US-2.3 — Язык и термины перед обработкой
**Как** пользователь, **когда** загружаю файл, **я хочу** чтобы язык определился автоматически (или задать вручную) и добавить известные имена/термины, **чтобы** распознавание было точнее.
**Приоритет:** MVP
- Автоопределение языка + ручной выбор в настройках; ввод известных имён/терминов до обработки.

## Эпик 3. Offline-first и синхронизация

### US-3.1 — Запись и работа без сети
**Как** пользователь, **когда** нет интернета, **я хочу** записывать и просматривать готовые протоколы/задачи, **чтобы** работа не зависела от связи.
**Приоритет:** MVP
- Без сети работают: запись, просмотр готовых протоколов/задач, редактирование локальных данных, отправка в Telegram/почту (в очередь).
- Транскрипция офлайн не делается (осознанное ограничение).

**Критерии приёмки:**
- Дано нет сети, когда пользователь записывает встречу, тогда она сохраняется локально со статусом «на устройстве» (одна галочка).
- Дано сеть появилась, когда происходит синхронизация, тогда статус меняется на «отправлено» (две галочки) и приходит уведомление.

### US-3.2 — Статусы синхронизации (галочки)
**Как** пользователь, **когда** записал офлайн, **я хочу** видеть, что сохранено только на устройстве, а что уже в облаке, **чтобы** не потерять запись.
**Приоритет:** MVP
- Одна галочка = только на устройстве; две галочки = отправлено. Раздел «Несинхронизированные записи».

### US-3.3 — Чанковая синхронизация при слабой сети
**Как** пользователь, **когда** сеть появляется на короткое время, **я хочу** чтобы файл догружался кусками, **чтобы** не начинать выгрузку заново.
**Приоритет:** MVP
- Разбиение на чанки; неотправленный чанк не считается доставленным; возобновление с места обрыва.

**Критерии приёмки:**
- Дано выгрузка прервалась, когда сеть вернулась, тогда загрузка возобновляется с недоставленного чанка, а не с нуля.

### US-3.4 — Управление синхронизацией и трафиком
**Как** пользователь, **когда** я на мобильном интернете, **я хочу** запустить синхронизацию вручную и запретить выгрузку по мобильной сети, **чтобы** экономить трафик.
**Приоритет:** MVP
- Кнопка «Синхронизация»; запрет выгрузки через мобильную сеть; предупреждение при нехватке памяти.

### US-3.5 — Облачное vs локальное хранение
**Как** пользователь, **когда** настраиваю приложение, **я хочу** выбрать, где хранить записи, **чтобы** контролировать доступ и сохранность.
**Приоритет:** MVP
- **По умолчанию — облако** (данные доступны с любого устройства после входа).
- Опция «только на устройстве» → при удалении приложения данные теряются (с предупреждением).
- Шифрование записи на устройстве — обязательно. Аудио/записи хранятся максимум 1 год.

### US-3.6 — Продолжение работы между устройствами
**Как** пользователь, **когда** начал на телефоне, **я хочу** продолжить на компьютере, **чтобы** работать с любого устройства.
**Приоритет:** MVP
- Встречи, протоколы, задачи и настройки синхронизируются между устройствами; запись, начатую на телефоне, можно продолжить на ПК.

**Критерии приёмки:**
- Дано запись начата на телефоне, когда пользователь открывает приложение на ПК, тогда встреча доступна и работу можно продолжить.

## Эпик 4. Обработка и статусы

### US-4.1 — Понятные статусы конвейера
**Как** пользователь, **когда** встреча обрабатывается, **я хочу** видеть простые статусы, **чтобы** понимать, что происходит.
**Приоритет:** MVP
- Показываем: загружается → расшифровывается → создаётся протокол → извлекаются задачи → готово. Технические детали не показываем.
- Процент — только для «загружается / расшифровывается». Ожидаемое время — не показываем.

**Критерии приёмки:**
- Дано встреча обрабатывается, когда пользователь открывает её, тогда виден текущий понятный статус (без технических терминов).

### US-4.2 — Параллельная работа во время обработки
**Как** пользователь, **когда** идёт обработка, **я хочу** пользоваться другими функциями и обрабатывать несколько встреч сразу, **чтобы** не ждать.
**Приоритет:** MVP
- Несколько встреч одновременно; приоритет — по времени запуска (FIFO). Обработку можно отменить.

### US-4.3 — Автовосстановление при сбое этапа
**Как** пользователь, **когда** транскрипция готова, а генерация протокола упала, **я хочу** чтобы система сама перезапустила упавший этап, **чтобы** не делать это вручную.
**Приоритет:** MVP
- Автоперезапуск генерации протокола; при появлении ошибки — кнопка «Перезапустить обработку». Повторно запускать отдельный этап вручную — нельзя.

**Критерии приёмки:**
- Дано транскрипт готов, когда генерация протокола падает, тогда система автоматически перезапускает генерацию.
- Дано ошибка повторилась более одного раза, тогда она считается критической и предлагается обращение в поддержку.

### US-4.4 — Сигнал о низком качестве аудио
**Как** пользователь, **когда** качество записи низкое, **я хочу** узнать об этом во время записи, **чтобы** успеть исправить условия.
**Приоритет:** MVP
- Сигнал о низком качестве в процессе записи.

---

## Эпик 5. Уведомления

### US-5.1 — Уведомление о готовности и сбоях
**Как** пользователь, **когда** обработка завершена или произошёл сбой, **я хочу** получить уведомление, **чтобы** сразу перейти к результату.
**Приоритет:** MVP
- Обязательные: завершение записи, готовность протокола, сбой/остановка. Отключаемые: промежуточные этапы.
- Каналы: in-app + Web Push + Telegram + Email. По умолчанию — Email. Дублирование — максимум в 2 канала.
- Из уведомления — переход сразу к результату. Режим «не беспокоить» — есть.

**Критерии приёмки:**
- Дано обработка завершена, когда готов протокол, тогда приходит уведомление в выбранный канал со ссылкой на результат.
- Дано произошёл сбой, тогда приходит уведомление о сбое с указанием встречи.

### US-5.2 — Содержание и статус уведомления
**Как** пользователь, **когда** получаю уведомление, **я хочу** видеть суть события, **чтобы** понять его без открытия приложения.
**Приоритет:** MVP
- Текст: «Встреча <название> распознана, задачи поставлены» / «По встрече <название> сбой».
- Прочитано = просмотрено в приложении или доставлено в TG/Email. История уведомлений — 90 дней.

## Эпик 6. Стенограмма

### US-6.1 — Просмотр стенограммы
**Как** пользователь, **когда** хочу проверить встречу, **я хочу** открыть текст стенограммы, **чтобы** прочитать, что было сказано.
**Приоритет:** MVP
- Открытие из карточки встречи, прокрутка. Спикеры — заголовками (имя или «Спикер N»).
- Фрагменты низкой уверенности — жёлтая подсветка. Слова-паразиты и повторы остаются.
- Два режима: дословный и очищенный текст.

### US-6.2 — Воспроизведение и навигация по аудио
**Как** пользователь, **когда** сомневаюсь в тексте, **я хочу** прослушать фрагмент с изменением скорости и перемоткой, **чтобы** сверить с записью.
**Приоритет:** MVP
- Изменение скорости; перемотка 5–15 сек; автоподсветка текущей реплики.
- Клик по таймкоду и жёсткая синхронизация текста с аудио — не в MVP.

### US-6.3 — Поиск, копирование, экспорт стенограммы
**Как** пользователь, **когда** ищу конкретный момент, **я хочу** искать по стенограмме, копировать фрагменты и экспортировать её, **чтобы** переиспользовать текст.
**Приоритет:** MVP
- Поиск по стенограмме; копирование фрагмента; экспорт отдельно от протокола (TXT/DOCX/MD).
- Комментарии, задача из выделения, ссылка на таймкод — не в MVP.

### US-6.4 — Редактирование стенограммы
**Как** пользователь, **когда** вижу ошибку распознавания, **я хочу** поправить текст и массово заменить имя/термин, **чтобы** стенограмма была точной.
**Приоритет:** MVP
- Правит пользователь; исходная AI-версия сохраняется; массовая замена термина/имени.
- Протокол после правки автоматически не перегенерируется — только вручную. Уже созданные задачи остаются.
- Можно заблокировать результат от дальнейшего редактирования.
- История изменений и отмена правок — не в MVP.

## Эпик 7. Спикеры и контакты

### US-7.1 — Разделение и переименование спикеров
**Как** пользователь, **когда** на встрече было несколько человек, **я хочу** чтобы речь разделилась по спикерам и я мог их переименовать, **чтобы** протокол был читаемым.
**Приоритет:** MVP
- Диаризация; если возможно — определение имени по обращениям, иначе «Спикер N».
- Рекомендация «Спикер 1 → имя», пользователь принимает/правит.
- Объединение ошибочно разделённых и разделение ошибочно объединённых спикеров.
- Индикатор уверенности определения спикера (шкала).
- Запоминание голоса между встречами — **отложено до юридической проверки** 152-ФЗ.

### US-7.2 — Импорт и сопоставление контактов
**Как** пользователь, **когда** назначаю исполнителей, **я хочу** импортировать контакты и сопоставлять их с именами из транскрипта, **чтобы** быстро ставить задачи нужным людям.
**Приоритет:** MVP
- Источники: телефонная книга (на телефоне), Email/Битрикс24 (на ПК), Telegram — по выбору.
- Обязательное поле — имя; организация/должность — если есть. Один основной email на контакт.
- Дубли — приложение спрашивает, объединять ли. При сопоставлении — предложение вариантов; при нескольких совпадениях выбирает пользователь.
- Контакты принадлежат пользователю; он может запретить доступ к контактам. Редактирует контакт только владелец.

## Эпик 8. Протокол встречи

### US-8.1 — Структурированный протокол по шаблону
**Как** пользователь, **когда** встреча обработана, **я хочу** получить структурированный протокол по выбранному шаблону, **чтобы** не собирать его вручную.
**Приоритет:** MVP
- Разделы (участники, повестка, обсуждения, решения, задачи, вопросы, следующие шаги) задаются шаблоном.
- Структура зависит от типа встречи (планёрка, переговоры, интервью, проектная, совещание, «свои мысли»).
- Шаблоны — пользовательские; есть шаблон по умолчанию. Корпоративные шаблоны — следующий этап.
- В заголовке: тема встречи + дата. Спорные решения (по мнению AI) — жёлтая подсветка.
- **Executive summary** — краткая выжимка на 3–5 строк в начале протокола. Краткая и полная версии.

**Критерии приёмки:**
- Дано выбран шаблон, когда протокол сгенерирован, тогда он содержит обязательные для шаблона разделы и executive summary.
- Дано AI отметил спорное решение, тогда оно подсвечено жёлтым.

### US-8.2 — Редактирование и повторная генерация протокола
**Как** пользователь, **когда** в протоколе есть неточности, **я хочу** отредактировать его (текстом или голосом) и при необходимости перегенерировать с инструкцией, **чтобы** получить корректный результат.
**Приоритет:** MVP
- Правка текстом и голосом; можно отмечать AI-текст и ручные изменения.
- Повторная генерация с инструкцией; старая версия затирается (версий не храним).
- Вставка изображений/таблиц/ссылок; приложения к протоколу; печатная версия.
- Совместное редактирование, комментарии, упоминания, история версий — не в MVP.
- Финал = подтверждён пользователем; можно закрыть от изменений. Экспорт: DOCX/PDF/MD.

---

## Эпик 9. Извлечение задач «кто-что-когда»

### US-9.1 — Обнаружение задач
**Как** пользователь, **когда** протокол готов, **я хочу** получить список задач в формате «кто-что-когда», **чтобы** ничего не потерять из договорённостей.
**Приоритет:** MVP
- Задача = дело с ответственным, сроком и образом результата.
- Обязательные поля: исполнитель, описание, срок; остальное — опционально.
- Задачи без исполнителя/срока извлекаются, в этих полях — «?».
- Одна реплика может дать несколько задач; похожие объединяются; дубли обнаруживаются.
- Уверенность AI и объяснение «почему это задача» — не показываем.

**Критерии приёмки:**
- Дано в речи поручение без срока, когда извлекаются задачи, тогда задача создаётся, а в поле «срок» стоит «?».
- Дано две похожие задачи, тогда предлагается их объединить.

### US-9.2 — Проверка и правка задач
**Как** пользователь, **когда** задачи извлечены, **я хочу** проверить и подтвердить их (в т.ч. массово), **чтобы** отправить корректный список.
**Приоритет:** MVP
- Подтверждает пользователь; режим «подтвердить все»; можно отправить без проверки.
- Обязательно проверяются: кто, что, когда. Массовое назначение исполнителя и изменение срока; удаление ошибочной и создание пропущенной задачи.
- Любой задаче присваивается ответственный. Приоритет — **precision** («хуже создать ложную задачу»).

### US-9.3 — Разрешение относительных дат
**Как** пользователь, **когда** в речи сказано «к пятнице», **я хочу** чтобы дата превратилась в конкретную, **чтобы** срок был однозначным.
**Приоритет:** MVP
- «к пятнице» → ближайшая пятница после встречи. «до конца недели» → пятница 17:00 этой недели. «в течение месяца» → 17:00 последнего рабочего дня месяца.
- Часовой пояс — из системы устройства. Праздники/выходные — не учитываем (если не задано).
- Преобразованную дату пользователь подтверждает; при неоднозначности — «?». Просроченная дата → уведомление.
- Повторяющиеся задачи — нужны; зависимости — не в MVP.

**Критерии приёмки:**
- Дано в речи «до конца недели», когда разрешается срок, тогда ставится пятница 17:00 текущей недели.
- Дано срок неоднозначен, тогда в поле срока «?» и запрашивается подтверждение.

### US-9.4 — Определение исполнителя
**Как** пользователь, **когда** в речи назван исполнитель, **я хочу** чтобы AI предложил его, а я подтвердил/поправил, **чтобы** задача ушла нужному человеку.
**Приоритет:** MVP
- «мы сделаем» → ответственный = говорящий. «пусть маркетинг подготовит» → руководитель подразделения.
- Несколько людей с одним именем → выбирает пользователь. Внешний исполнитель — можно; без аккаунта — поле подсвечивается, задача уходит по Email/Telegram.

## Эпик 10. Проекты, поиск

### US-10.1 — Проекты
**Как** пользователь, **когда** встречи связаны одной темой, **я хочу** объединить их в проект, **чтобы** видеть их и задачи вместе.
**Приоритет:** MVP
- Проект = раздел, объединяющий встречи по теме. Обязательное поле — тема; остальное опционально.
- Владелец — пользователь; могут быть участники. Встреча может относиться к нескольким проектам; проект можно архивировать.
- Автоопределение проекта — не нужно. **Удаление проекта → встречи переходят в «Без проекта»** (не удаляются; ADR-030). Подпроекты — не в MVP.
- Сводка по проекту: список встреч, открытые задачи, хронология решений, счётчики. Хронология решений — лента «дата → решение → встреча».
- **Группы доступа по проектам — не в MVP** (появятся с командной версией).

### US-10.2 — Поиск и фильтрация
**Как** пользователь, **когда** ищу прошлую договорённость, **я хочу** искать по задачам и фильтровать встречи, **чтобы** быстро находить нужное.
**Приоритет:** MVP
- Ищем: ответственного, сроки, текст/содержание задачи. Фильтры: дата, проект, участник, статус, наличие задач, источник, тип встречи.
- Голосовой поиск — да; переход к конкретной встрече — да. Экспорт результатов — да.
- Поиск по содержимому стенограммы, **семантический поиск**, сохранённые фильтры, история поиска — не в MVP.

## Эпик 11. Отправка и интеграции

### US-11.1 — Общие правила отправки
**Как** пользователь, **когда** протокол готов, **я хочу** отправить протокол/резюме/задачи выбранным получателям, **чтобы** разослать договорённости.
**Приоритет:** MVP
- Отправляем: протокол, краткое резюме, задачи. Получателей выбирает пользователь; AI может предложить их по истории.
- Подтверждение и предпросмотр — по настройке; можно настроить автоотправку.
- Ссылка на результат: срок действия 1 неделя, доступ можно отозвать; пароль — не нужен.
- Фиксируем доставку; при ошибке канала — сообщение + альтернативный канал; автоповтор 3 раза, затем ошибка.

**Критерии приёмки:**
- Дано ошибка отправки в один канал, когда исчерпаны 3 попытки, тогда показывается ошибка и предлагается сменить канал.
- Дано создана ссылка на результат, тогда через неделю она перестаёт действовать.

### US-11.2 — Email
**Как** пользователь, **когда** рассылаю протокол, **я хочу** отправить письмо с вложением, **чтобы** участники получили итог встречи.
**Приоритет:** MVP
- Письмо **только от имени VoxMate** (корпоративный SMTP не даём); тема/текст формируются автоматически.
- Вложение PDF/Word; шаблоны писем — есть; CC/BCC — опционально.
- Недействительный адрес → сообщение + предложение выбрать другой.

### US-11.3 — Telegram
**Как** пользователь, **когда** хочу быстро донести задачи, **я хочу** отправить их в личный чат исполнителю, **чтобы** он сразу их увидел.
**Приоритет:** MVP
- Отправка в личный чат; получатели — из контактов с привязанным Telegram.
- Если пользователь не найден — сообщение, оставить как есть.
- **Inline-кнопки** («Выполнено», «Взять в работу») — есть; отметка задачи выполненной в Telegram — возможна; обратная синхронизация статуса — не обязательна.

### US-11.4 — Битрикс24
**Как** пользователь, **когда** веду задачи в Б24, **я хочу** создавать там задачи из встречи, **чтобы** не дублировать вручную.
**Приоритет:** MVP
- Подключение — пользователем (в персональной версии). Сопоставление сотрудника Б24 — предлагается пользователю; если не найден — сообщение, задачу не отправляем.
- Группа проекта — выбирает пользователь; сохраняем ссылку на созданную задачу. Дубли → сообщение, решает пользователь.
- **Источник истины после отправки — Б24**: задача в VoxMate после отправки не меняется; изменения в Б24 обратно **не** синхронизируем.

## Эпик 12. Zero-click и автоматизация

### US-12.1 — Правила автоматизации
**Как** пользователь, **когда** сценарий повторяется, **я хочу** задать правило, **чтобы** не повторять действия вручную.
**Приоритет:** След. релиз
- AI без подтверждения: распознавание, черновой протокол и задачи. Всегда с подтверждением: финальный протокол, задачи, отправка.
- AI может предложить: название, проект, участников, задачи, получателей, отправку.
- Правила настраивает пользователь; есть справочник шаблонов и правил. Конфликт правил → выбор приоритетного или новое.
- Автодействие можно отменить; журнал автодействий ведём. Подтверждение запрашивается при уверенности распознавания ниже 90%.

**Критерии приёмки:**
- Дано задано правило автоотправки, когда встреча ему соответствует, тогда действие выполняется и попадает в журнал.
- Дано уверенность распознавания ниже 90%, тогда запрашивается подтверждение.

## Эпик 13. Главный экран и управление

### US-13.1 — Контекстный главный экран
**Как** пользователь, **когда** открываю приложение, **я хочу** сразу видеть главное действие и статус текущих встреч, **чтобы** быстро продолжить работу.
**Приоритет:** MVP
- Первое действие — Запись. Утром — последние задачи, дата, кнопка новой встречи. После встречи — статус обработки.
- Приоритет блоков: текущая обработка → готовые протоколы → задачи → предстоящие встречи → ошибки.
- Просроченные задачи — красным. Показываем ~10 карточек; экран настраивается. Конфиденциальные данные на экране блокировки — не показываем.

### US-13.2 — Жесты и голосовое управление
**Как** пользователь, **когда** работаю одной рукой, **я хочу** типовые действия жестами и голосом, **чтобы** управлять быстрее.
**Приоритет:** MVP (базовые) / расширенные — След. релиз
- Свайпами — типовые действия. Опасные (удаление, отправка) — с подтверждением.
- Голосом: команды, правка текста, поиск, создание задачи (опция). Индикация «слушаю» — подсветка + сигнал.
- Без интернета голосом: запись и работа с сохранёнными данными.

## Эпик 14. Telegram standalone-бот

### US-14.1 — Обработка через бота
**Как** пользователь, **когда** мне удобнее в Telegram, **я хочу** прислать боту голосовое/аудио/видео и получить протокол и задачи, **чтобы** не открывать приложение.
**Приоритет:** MVP (ранняя проверка гипотезы)
- Материалы: голосовое, аудио, видео, ссылка. Бот: подтверждает получение → сообщает об обработке → выдаёт протокол + задачи.
- Выбор простого формата результата; можно отправить результат другому и привязать к проекту.
- Связка с аккаунтом — авторизация. Только **личные чаты**; групповые — следующий этап. Синхронизация истории бота с приложением — да.
- Работает **без установки PWA**.

**Критерии приёмки:**
- Дано пользователь прислал боту голосовое, когда обработка завершена, тогда бот возвращает протокол и список задач.
- Дано файл превышает лимит Telegram, тогда бот сообщает, что файл слишком большой.

## Эпик 15. Бот-участник конференций

### US-15.1 — Подключение бота к конференции
**Как** пользователь, **когда** у меня онлайн-встреча, **я хочу** пригласить бота по ссылке, **чтобы** он записал встречу без моего участия в записи.
**Приоритет:** Поздний этап
- Организатор шлёт ссылку боту; бот подключается за ~10 сек и сообщает о подключении. В списке участников — «Бот-ассистент»; объявляет о записи.
- Комната ожидания → ждёт, не пишет; недопуск → сообщение пользователю. Пока только аудио; список участников — да.
- Разрыв связи → пауза и продолжение; бот переподключается. Остановить может только организатор; удаление бота → уведомление.
- Технический журнал подключений — ведём. Автоссылки из календаря — не в MVP.

## Эпик 16. Аккаунт, настройки, безопасность

### US-16.1 — Регистрация и вход
**Как** новый пользователь, **когда** начинаю работу, **я хочу** войти удобным способом, **чтобы** быстро начать.
**Приоритет:** MVP
- Входы (все шесть в MVP, ADR-027): Google, Сбер ID, Яндекс, ВКонтакте, email+пароль, Telegram.
- Без регистрации — простые действия (короткие записи до ~1 мин); дальше регистрация обязательна.
- Совпадение email → сообщение «уже зарегистрирован»; объединение аккаунтов — подтверждением.
- Восстановление доступа: email-ссылка + вход через привязанный OAuth. Список устройств и уведомление о новом входе — да.
- 2FA — не в MVP; **SSO — не в MVP**. Удаление аккаунта — есть; архив данных по закону, но не более 1 года.

### US-16.2 — Пользовательские настройки
**Как** пользователь, **когда** настраиваю приложение, **я хочу** управлять языком, шаблонами, каналами и хранением, **чтобы** подстроить продукт под себя.
**Приоритет:** MVP
- Язык интерфейса — русский; языки аудио — русский, английский; автоопределение.
- Шаблон протокола по умолчанию; запоминание канала отправки; настройки уведомлений; сброс персонализации AI.
- Настройки открываются одинаково на всех устройствах. Пользовательский словарь, правила автоназвания, упрощённый режим — не в MVP.

### US-16.3 — Согласие на запись (152-ФЗ)
**Как** пользователь, **когда** записываю встречу с участниками, **я хочу** чтобы приложение помогло с уведомлением о записи, **чтобы** соблюсти закон.
**Приоритет:** MVP
- Предупреждение перед каждой записью. Уведомление участников и хранение доказательства — по требованию закона РФ (**юр. проверка**).
- Отказ участника → выбор: записать всё равно / без участника / отказаться.
- Отдельное согласие на запись/транскрипцию/AI-анализ/трансграничную передачу — по требованию закона.

### US-16.4 — Доступ, хранение и удаление
**Как** пользователь, **когда** работаю с чувствительными встречами, **я хочу** управлять доступом и сроками хранения, **чтобы** контролировать данные.
**Приоритет:** MVP
- Аудио/стенограмму видит тот, кто записал; протокол/задачи — участники. Метка «конфиденциально»; запрет скачивания; уведомление владельцу о просмотре.
- Поддержка видит содержимое только с согласия пользователя.
- Протоколы/задачи — без ограничения; аудио/записи — максимум 1 год. Корзина — 1 неделя, затем окончательное удаление (включая резервные копии); отчёт об удалении.

### US-16.5 — Интеграция с календарём (Outlook / Microsoft 365)
**Как** пользователь, **когда** у меня запланирована встреча, **я хочу** чтобы приложение напомнило начать запись, **чтобы** не забыть её записать.
**Приоритет:** MVP
- Календарь MVP — **Outlook / Microsoft 365** через Microsoft OAuth (ADR-028). Чтение разрешённых событий, минимизация scopes, refresh token.
- Напоминание по умолчанию за 15 минут; кнопка из напоминания открывает карточку встречи и позволяет запустить запись.
- Изменение события в календаре обновляет напоминание; истёкший токен → предложение переподключить.
- Календарь и вход — разные экосистемы: Microsoft OAuth (календарь) и провайдеры входа (Яндекс/Telegram/…) настраиваются раздельно.

**Критерии приёмки:**
- Дано подключён Outlook, когда до встречи 15 минут, тогда приходит напоминание с кнопкой начать запись.
- Дано событие в календаре изменилось, тогда напоминание обновляется.
- Дано истёк Microsoft OAuth token, тогда пользователю предлагается переподключить календарь.

## Эпик 17. Ошибки и нестандартные ситуации

### US-17.1 — Каталог исключений
**Как** пользователь, **когда** что-то идёт не так, **я хочу** понятное сообщение и понятный следующий шаг, **чтобы** не застрять.
**Приоритет:** MVP

| Ситуация | Реакция |
|----------|---------|
| Нет доступа к микрофону / отключён / нет звука | Сообщение, запись не начинается, предложение включить микрофон, алерт пользователю |
| Очень тихо / сильный шум | Предупреждение; если согласен — запись продолжается |
| Мало места на устройстве | Заблаговременное предупреждение (если места хватит менее чем на ~15 мин) |
| Закрылся браузер / перезагрузка / встреча удалена другим | Без сообщений |
| Пропала сеть / оборвалась загрузка / файл загружен частично | Сообщение; загрузка возобновляется при связи |
| Файл повреждён / кодек не поддерживается | Сообщение + просьба загрузить корректный файл |
| ASR/LLM недоступны | Сообщение «приложение пока не может обработать» |
| Стенограмма готова частично | Без сообщения, система продолжает распознавание |
| Не определён язык | По умолчанию русский |
| Не определён спикер | «Спикер N» |
| Не извлечены задачи | В протоколе: «задачи извлечь не удалось» |
| Не определён исполнитель/срок | «?» + просьба заполнить вручную |
| Не создана задача в Б24 | Предупреждение |
| Telegram-бот заблокирован | Сообщение + кнопка «обратиться к администратору» |
| Email не доставлен | Предупреждение |
| Истёк токен интеграции | Сообщение об исчерпании лимита / когда восстановится |
| Нет прав | Сообщение «прав нет» |
| Опасная ошибка в результате AI | Предупреждение в тексте результата |
| Случайно удалена встреча | Восстановление из корзины (доступна 1 неделю) |

## Эпик 18. Устройства, доступность, поддержка

### US-18.1 — Поддержка устройств
**Как** пользователь, **когда** работаю с телефона или ПК, **я хочу** чтобы приложение адаптировалось под устройство, **чтобы** было удобно на любом экране.
**Приоритет:** MVP
- Приоритет: Android-телефон + Windows-ПК; поддержка iOS и планшета. Формат: PWA + браузер.
- Минимальные ОС: Windows 10, Android 10+, iOS 15+. **Windows-браузер MVP — только Google Chrome** (ADR-029); Edge/Firefox — post-MVP. Структура экрана различается на телефоне и ПК; телефон — только вертикальная ориентация.
- Горячие клавиши на ПК; **drag-and-drop файлов на ПК — есть**. Несколько окон, foldable — не в MVP.

### US-18.2 — Доступность
**Как** пользователь, **я хочу** базовую доступность, **чтобы** пользоваться приложением было комфортно.
**Приоритет:** MVP (базовый) / расширенный — След. релиз
- Управление мышью, клавиатурой, пальцами; индикаторы звука. Голосовой запуск записи (альтернатива удержанию кнопки).
- Соблюдаем quality floor из техспеки: контраст WCAG AA, видимый keyboard-focus, `prefers-reduced-motion`.
- Screen reader, режим контраста, масштаб 200%, haptic-настройка — **отложены** (след. релиз).

### US-18.3 — Обратная связь и поддержка
**Как** пользователь, **когда** вижу ошибку или хочу оценить результат, **я хочу** сообщить о проблеме и поставить оценку, **чтобы** качество росло.
**Приоритет:** MVP
- Кнопка «сообщить об ошибке» (стенограмма/пропущенная задача). Оценка результата 1–5 + опциональный комментарий; периодический опрос (напр. каждая 10-я встреча).
- Разрешение использовать запись для улучшения — по согласию. Чат поддержки + статус обращений; уведомление, когда ошибка исправлена.

### US-18.4 — Административная панель
**Как** администратор продукта, **когда** слежу за работой сервиса, **я хочу** видеть статусы и управлять обработкой, **чтобы** поддерживать качество.
**Приоритет:** След. релиз
- Статус обработки без содержимого; перезапуск задач; длина очереди; стоимость обработки.
- Блокировка пользователя и ограничение подозрительной активности; журнал ошибок интеграций. Смена AI-модели без деплоя; восстановление удалённой встречи.
- Поддержка не видит конфиденциальные данные и содержимое аудио/файлов.

## Разобранные точечные вопросы (бывшая Часть 4)

| Вопрос | Решение |
|--------|---------|
| Группы доступа по проектам | Нет в MVP (появятся с командной версией) |
| Внешний подрядчик — доступ к одному проекту | Нет в MVP (следствие персональной модели) |
| Кто редактирует контакт | Только владелец-пользователь |
| Разные email для разных каналов | Не нужны; один основной email на контакт |
| Синхронизация статуса из Б24 обратно | Не синхронизируем; Б24 — конечная точка |
| Источник истины после отправки задачи | Б24; в VoxMate задача после отправки не меняется |
| Хранение по умолчанию | Облако (с опцией «только устройство») |
| Биометрия голоса | На паузе до юридической проверки 152-ФЗ |

## Осталось для полной готовности бэклога

1. **Юридическая проверка 152-ФЗ:** биометрия голоса, требования к согласию (запись/транскрипция/AI-анализ/трансграничная передача). Единственный внешний блокер.
2. **Замер целевых метрик AI** на реальных встречах (подтвердить precision ≥ 0.8, recall ≥ 0.7).
3. При переходе к командной версии — вернуть отложенные истории (роли, организации, общий доступ, корпоративные шаблоны, группы доступа).

Продуктовых противоречий не осталось — бэклог готов к детализации и оценке.
