## 5. Реальная аудио-диаризация (WhisperX + pyannote)

Пользователь отметил, что в одной из записей ~10 разных спикеров, а текстовое угадывание находит только 2–3. Исследовали:

- Через официальную документацию OpenRouter подтвердили: ни одна ASR-модель на OpenRouter не отдаёт таймкоды или диаризацию — только `{ text, usage }`. Значит реальная диаризация в принципе невозможна поверх OpenRouter.
- Решили поднять отдельный self-hosted сервис на **WhisperX** (faster-whisper + forced alignment + pyannote diarization) — новый сервис `diarizer/` (Python, FastAPI) в `docker-compose.yml`, вызываемый воркером с автоматическим фолбэком на старую текстовую схему при любой ошибке.

### Долгая цепочка отладки сборки diarizer-образа
1. `libcudart.so.13` — конфликт CPU/GPU сборок torch/torchaudio → зафиксировали версии из `--index-url .../whl/cpu`.
2. `cannot enable executable stack` у ctranslate2 → пакета `execstack` нет в Debian trixie, использовали `patchelf --clear-execstack`.
3. `ModuleNotFoundError: matplotlib` → добавили в requirements.
4. torch 2.6 включил `weights_only=True` по умолчанию → чекпоинт pyannote с omegaconf не грузится → монki-патч `torch.load` внутри `app.py` (форсируем `weights_only=False`, источник моделей доверенный — свой HF-токен).
5. `huggingface_hub` слишком новый, убрал `use_auth_token` → зафиксировали `huggingface_hub==0.24.6`.
6. HF-токен не имел доступа к гейтед-репозиториям (`403 ... fine-grained token settings`) → пользователь включил разрешение «Read access to contents of all public gated repos».
7. Отдельная лицензия `pyannote/segmentation-3.0` (суб-зависимость `speaker-diarization-3.1`) — пользователь принял её отдельно.
8. Успешный тест на маленьком файле: `{"language":"ru","segments":[{"start":1.467,"end":6.182,"speaker":"SPEAKER_00","text":"..."}]}`.

### Проблема на большом файле (реальная встреча, ~10 спикеров)
- Node `fetch` по умолчанию (undici) обрывает соединение через 5 минут (`headersTimeout`/`bodyTimeout`) — обработка большого файла на CPU занимает намного дольше, из-за чего конвейер откатывался на старую схему, а диаризатор продолжал впустую работать в фоне.
- Добавили зависимость `undici` в `backend/package.json` и явно отключили таймауты для воркера (`setGlobalDispatcher(new Agent({ headersTimeout: 0, bodyTimeout: 0 }))`), т.к. это доверенный внутренний сервис.
- Поймали ещё один Docker-нюанс: анонимный volume для `node_modules` пережил пересборку образа и не подхватил новую зависимость — пересоздали с `--force-recreate -V`.
- Финальный прогон на реальном ~10-спикерном файле шёл 48+ минут на CPU (без GPU) и был остановлен пользователем вручную («останови процесс») — конвейер и инфраструктура рабочие, просто CPU-инференс на большом аудио объективно медленный.
