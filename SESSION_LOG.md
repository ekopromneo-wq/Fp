## 5. Реальная аудио-диаризация (WhisperX + pyannote)

Пользователь отметил, что в одной из записей ~10 разных спикеров, а текстовое угадывание находит только 2–3. Исследовали:

- Через официальную документацию OpenRouter подтвердили: ни одна ASR-модель на OpenRouter не отдаёт таймкоды или диаризацию — только `{ text, usage }`. Значит реальная диаризация в принципе невозможна поверх OpenRouter.
- Решили поднять отдельный self-hosted сервис на **WhisperX** (faster-whisper + forced alignment + pyannote diarization) — новый сервис `diarizer/` (Python, FastAPI) в `docker-compose.yml`, вызываемый воркером с автоматическим фолбэком на старую текстовую схему при любой ошибке.

### Долгая цепочка отладки сборки diarizer-образа
1. `libcudart.so.13` — конфликт CPU/GPU сборок torch/torchaudio → зафиксировали версии из `--index-url .../whl/cpu`.
2. `cannot enable executable stack` у ctranslate2 → пакета `execstack` нет в Debian trixie, использовали `patchelf --clear-execstack`.
3. `ModuleNotFoundError: matplotlib` → добавили в requirements.
4. torch 2.6 включил `weights_only=True` по умолчанию → чекпоинт pyannote с omegaconf не грузится → монki-патч `torch.load` внутри `app.py` (форсируем `weights_only=False`, источник моделей доверенный — свой HF-токен).
5. `huggingface_hub` слишком новый, убрал `use_auth_token` → зафиксировали `huggingface_hub==0.24.6`.
6. HF-токен не имел доступа к гейтед-репозиториям (`403 ... fine-grained token settings`) → пользователь включил разрешение «Read access to contents of all public gated repos».
7. Отдельная лицензия `pyannote/segmentation-3.0` (суб-зависимость `speaker-diarization-3.1`) — пользователь принял её отдельно.
8. Успешный тест на маленьком файле: `{"language":"ru","segments":[{"start":1.467,"end":6.182,"speaker":"SPEAKER_00","text":"..."}]}`.

### Проблема на большом файле (реальная встреча, ~10 спикеров)
- Node `fetch` по умолчанию (undici) обрывает соединение через 5 минут (`headersTimeout`/`bodyTimeout`) — обработка большого файла на CPU занимает намного дольше, из-за чего конвейер откатывался на старую схему, а диаризатор продолжал впустую работать в фоне.
- Добавили зависимость `undici` в `backend/package.json` и явно отключили таймауты для воркера (`setGlobalDispatcher(new Agent({ headersTimeout: 0, bodyTimeout: 0 }))`), т.к. это доверенный внутренний сервис.
- Поймали ещё один Docker-нюанс: анонимный volume для `node_modules` пережил пересборку образа и не подхватил новую зависимость — пересоздали с `--force-recreate -V`.
- Финальный прогон на реальном ~10-спикерном файле шёл 48+ минут на CPU (без GPU) и был остановлен пользователем вручную («останови процесс») — конвейер и инфраструктура рабочие, просто CPU-инференс на большом аудио объективно медленный.

## 4. Решение: полностью свой (self-hosted) рекордер, без сторонних ботов

Пользователь явно попросил: захватывать звонки из Zoom/Meet/Yandex/Bitrix24 без каких-либо третьих сервисов; Speech2Text/Shopot/Gemini использовать только для расшифровки уже готовой записи.

- Исследовали Bitrix24 REST API вебхука `bds` (пользователь-«бот для совещаний», создан пользователем в `bitrix.ekoprom.org`): среди 126 доступных методов **нет `im.call.*`** — то есть узнать о входящем звонке через обычный REST-опрос нельзя в принципе. Чтобы бот сам «ответил» на звонок (без этого запись не запускается), нужен либо постоянно открытый залогиненный браузер, либо публичный HTTPS-эндпоинт для `event.bind` (не проверено, есть ли подходящее событие) — оба варианта существенно тяжелее, чем для остальных платформ.
- По решению пользователя **Bitrix24 отложен**; начали с **Yandex Telemost** как самого простого случая (гостевая ссылка, без логина, без постоянного демона).

### Надёжное определение конца звонка

Первая версия ждала конкретный текст на странице — ненадёжно. Заменили на комбинацию сигналов (`src/endDetection.js`):
1. **WebRTC connection state** (патчим `window.RTCPeerConnection` через `page.addInitScript`, см. `webrtcMonitor.js`) — общий сигнал для любой платформы на WebRTC, с debounce 15 сек на случай кратких обрывов.
2. **Длительная тишина** в записи (`ffmpeg silencedetect`) — если тишина держится >90 сек (и звонок шёл минимум 45 сек).
3. Текст на странице (платформо-специфичный, как резервный сигнал).
4. Жёсткий потолок 4 часа на случай, если все остальные сигналы промолчали.
5. Ручная остановка через `/jobs/:id/stop` (кнопка «Остановить бота» в интерфейсе) остаётся всегда доступна.

## Новый способ диаризации: Whisper + Gemini + Claude (2026-07-09/продолжение)

Пользователь остался недоволен качеством существующих методов диаризации и предложил pipeline с разделением ролей между тремя моделями вместо одной, которая должна одновременно расшифровывать, разделять по спикерам и угадывать имена.

### Архитектура (`backend/src/pipelineDiarizer.js`, метод `pipeline`)

1. Нормализация аудио в моно 16kHz mp3 (один раз, ffmpeg).
2. Нарезка **только для Whisper** на чанки по `PIPELINE_CHUNK_SECONDS` (по умолчанию 480с/8мин) — решили НЕ чанковать голосовую диаризацию (см. ниже почему).
3. `openai/whisper-large-v3` через OpenRouter `/api/v1/audio/transcriptions` с `response_format: 'verbose_json'` + `timestamp_granularities: ['segment']` — даёт точный текст с реальными таймкодами. **Важное открытие сессии**: предыдущая запись в этом логе (04.07) о том, что «ни одна ASR-модель на OpenRouter не отдаёт таймкоды», оказалась верна только для дефолтного `response_format`; с `verbose_json` таймкоды отдаются (подтверждено через доки OpenRouter и живой прогон). Работает только если запрос попадает на OpenAI-совместимого провайдера (OpenAI/Groq/Together) — без явного пиннинга провайдера, полагаемся на автороутинг OpenRouter.
4. `google/gemini-2.5-pro` (по выбору пользователя, а не `gpt-audio`) — слушает **весь файл целиком одним вызовом** (не чанкуется), отдаёт только грубые интервалы `{speaker, start, end}` по голосу, без текста — текст ей не доверяем, он уже есть от Whisper. Решение не чанковать этот шаг было осознанным выбором пользователя: чанковать — точнее для очень длинных записей, но потребовало бы сверки/склейки S1/S2/S3-меток между чанками (один и тот же голос в разных чанках получает разные локальные метки); без чанкования эта проблема просто не возникает, метки глобально консистентны сразу.
5. Склейка: каждый точный Whisper-сегмент назначается тому голосовому интервалу Gemini, с которым у него наибольшее пересечение по времени (fallback — ближайший интервал по расстоянию между серединами, если прямого пересечения нет).
6. `anthropic/claude-sonnet-5` — получает уже готовую стенограмму с голосовыми кластерами (S1/S2/...) и таймкодами, определяет реальные имена по самопредставлению/обращениям/лексике, с уровнем уверенности (`high`/`medium`/`low`) и цитатой-доказательством. Кластеры с `high`/`medium` уверенностью получают реальное имя, остальные — стабильную метку «Спикер N».
7. Финальный текст: `[MM:SS] Имя: реплика` — ровно то, что просил пользователь. Формат `{text, segments, speakerCount}` не изменился, поэтому **не потребовалось никаких изменений в БД или на фронтенде**, кроме одного пункта в выпадающем списке способа диаризации — `recording_speakers`/`mergeSpeakers` подхватывают новые лейблы автоматически, т.к. работают по факту наличия строки в `segment.speaker`.

Метод добавлен аддитивно (`DIARIZATION_METHODS` в `auth.js`) — существующие 4 способа не тронуты, ничей сохранённый выбор не изменился. Настройки моделей (`PIPELINE_WHISPER_MODEL`, `PIPELINE_VOICE_MODEL`, `PIPELINE_IDENTITY_MODEL`, `PIPELINE_CHUNK_SECONDS`, `PIPELINE_WHISPER_CONCURRENCY`) — только через переменные окружения, в UI сознательно не выведены (в отличие от Gemini/Kimi, где есть по одному полю модели) — весь пайплайн работает на одном общем `OPENROUTER_API_KEY`, отдельных пользовательских секретов не требует, а плодить 3 новых поля в settings-панели не хотелось.

По ходу нашли и убрали тройное дублирование кода между `openrouterAsr.js`/`geminiDiarizer.js`/`videoAudio.js` (обёртка над `spawn('ffmpeg', ...)`, определение формата файла по расширению) и двойное (разбор JSON-ответа LLM с обрезкой ```json-обёртки в `geminiDiarizer.js`/`speakerSplit.js`) — вынесли в `backend/src/ffmpeg.js` и `backend/src/llmJson.js` соответственно, новый пайплайн и все старые методы используют общий код.

## Аудит безопасности (та же сессия, коммит `71a5ef6`)

Пользователь попросил полный аудит безопасности бэкенда. Разбили на 3 параллельных направления (auth/доступ, инъекции/файлы, секреты/CORS/зависимости), самые серьёзные находки перепроверили руками перед фиксом.

**Найдено и исправлено:**
1. **CORS (критично)** — `index.js` безусловно отражал заголовок `Origin` запроса обратно в `Access-Control-Allow-Origin` **и одновременно** ставил `Access-Control-Allow-Credentials: true`. Поскольку сессия — httpOnly-кука, а фронтенд шлёт запросы с `credentials: 'include'`, это давало **любому стороннему сайту** возможность дёргать API от имени залогиненного пользователя его же куками и читать ответ — классическая дыра «reflect-any-origin + credentials». Заменили на явный allowlist (`CORS_ALLOWED_ORIGINS`, по умолчанию — локальный фронтенд); заголовки теперь ставятся только для origin из списка.
2. **`drizzle-orm` (0.38.4, high-severity SQL-injection advisory)** — обнаружили, что пакет вообще нигде не импортируется (`db.js` работает на чистом `pg`) — мёртвый груз от исходной спеки (см. запись от 04.07 про отход от tRPC/Drizzle). Заодно нашли и удалили `@trpc/server` — тоже нигде не используется. `npm audit` теперь: 0 уязвимостей.
3. **Загрузка аудио без лимита размера** (`POST /api/recordings/:id/audio`) — добавили `hono/body-limit`, порог намеренно щедрый (2 ГБ, `MAX_UPLOAD_BYTES`), т.к. в проекте реально бывают многочасовые видео-загрузки на десятки МБ — цель не ужать нормальную загрузку, а отсечь абсурдные/DoS-пейлоады.
4. **URL встречи для стороннего бота (Speech2Text) не проверялся на домен** — в отличие от собственного self-hosted recorder-bot (там allowlist уже был и работал), путь через `createMeetingBotRecording` принимал произвольный `meetingUrl` без проверки. В `meetingBot.js` уже был готовый `isSupportedMeetingUrl`/`SUPPORTED_PLATFORM_HOSTS`, но нигде не вызывался — подключили.

**Проверено вживую**: `curl -X OPTIONS` с чужим `Origin` теперь не получает вообще никаких CORS-заголовков (легитимный origin — получает); логин/сессия работают как прежде; `https://evil.example.com/...` в форме «Пригласить бота» отбивается 400 до обращения к Speech2Text, а `meet.google.com` по-прежнему проходит (тестовую запись из БД удалили после проверки).

**Осталось не исправлено (не просили в этой сессии, но стоит иметь в виду):** `/api/auth/register` без rate-limit/капчи/подтверждения email — можно наплодить аккаунты; порт API (4000) публикуется в docker-compose наружу, `/api/internal/recorder-bot/callback` защищён только секретом (не constant-time сравнение), а не сетевой изоляцией.

Это самый медленный и дорогой из 5 методов (Whisper по чанкам + Gemini на весь файл + Claude) — уместен как опция «для важных встреч», не как дефолт. Если для очень длинных/многолюдных записей (10+ голосов, 1+ час) точности голосовой диаризации Gemini на весь файл одним вызовом начнёт не хватать — есть заранее продуманный, но не реализованный путь: чанковать и шаг 4 тоже, добавив Claude на шаге 6 задачу сверки голосовых кластеров между чанками, а не только именования (обсуждали и осознанно отложили как более сложный и рискованный первый заход).

## Пайплайн диаризации: фикс на реальном длинном файле (та же сессия, следом)

Пользователь сам запустил пайплайн на реальной ~56-минутной записи (не тестовой) — и он упал с откатом на резерв. Разобрали причину и почин или сразу:

- **Баг**: Gemini на шаге голосовой диаризации иногда возвращает не совсем валидный JSON — строки в одинарных кавычках (`'S38'` вместо `"S38"`) вместо строгого JSON, несмотря на `response_format: {type:'json_object'}`. Строгий `JSON.parse` падал, весь пайплайн корректно откатывался на старый резервный метод (штатно, ничего не зависло), но сам новый метод не срабатывал.
- **Второй, более интересный симптом внутри той же ошибки**: судя по метке `S38`, Gemini на этой длинной записи насчитала 38 голосовых кластеров — явное переразбиение одних и тех же голосов, а не 38 реальных людей.
- **Фикс 1** — `parseIntervalsJson()` в `pipelineDiarizer.js`: при неудаче строгого парсинга повторяет попытку с заменой одинарных кавычек на двойные. Сделано **только для ответа голосовой диаризации** (там в схеме только метки спикеров и числа, живого текста нет) — специально не трогали общий `parseJsonObject` в `llmJson.js`, т.к. остальные вызовы (идентификация имён, geminiDiarizer, speakerSplit) возвращают настоящую прозу, где слепая замена кавычек могла бы испортить реальные апострофы.
- **Фикс 2** — усилили `VOICE_SYSTEM_PROMPT`: явно попросили не заводить новую метку без уверенности, что это другой голос, и подсказали, что у реальных встреч редко бывает больше ~15 разных говорящих (по выбору пользователя — только подсказка в промпте, без автоматического жёсткого потолка/отбраковки результата).
- **Повторный прогон на той же записи**: пайплайн отработал **без отката** — 393 сегмента Whisper, 321 интервал Gemini (с первой попытки), итог — 86 реплик на 15 спикеров (было 38). Один голос снова корректно и стабильно на всех репликах получил имя «Николай» — по фразе «Итак, Николай, начинай», привязка сработала на весь кластер, а не только на одну реплику.
- Коммит `0a9698c`, запушено.

- Аудио отдаётся фронтенду только проксировано через API (`GET /api/recordings/:id/audio`), MinIO никогда не отдаёт presigned URL в браузер — значит MinIO не нужно светить наружу, публичным должен быть только API.
- `CORS_ALLOWED_ORIGINS` (`backend/src/index.js`) — точный список origin через запятую, без wildcard.

### Деплой на сервере

## US-1.3: запись при свёрнутом приложении

Тот самый смежный пробел, отложенный при US-1.2. Здесь пришлось сразу зафиксировать границу возможного: голая PWA не может гарантированно пережить убийство вкладки ОС при нехватке памяти — это требует нативного foreground-сервиса (вне охвата), и явное «теперь запись никогда не прервётся» было бы преувеличением. Сделано то, что реально даёт веб-платформа:

- **Media Session API** (`useMicRecorder.js`): `navigator.mediaSession.playbackState` синхронизирован с состоянием записи (`playing`/`paused`/`none`), метаданные («Идёт запись встречи» / VoxMate) выставляются на старте. `pause`/`play`/`stop` повешены как action-хендлеры — это даёт системное медиа-уведомление на платформах, которые его поддерживают (пользователь видит и может управлять записью, не открывая вкладку/приложение), и вдобавок сигнализирует браузеру, что страница ведёт активную медиа-работу — по документации Chrome такие страницы освобождены от агрессивного tab discarding.
- **Wake Lock переподхватывается при возврате видимости**: по спеке браузер сам отпускает Wake Lock, как только страница скрывается — этого нельзя предотвратить, но раньше при возврате в приложение экран мог продолжать засыпать, потому что ничего не запрашивало lock заново. Теперь `visibilitychange` → `visible` при активной записи повторно запрашивает.
- **Предупреждение перед закрытием (`beforeunload`)** во время активной/приостановленной записи — страхует от самого частого реального способа потерять запись: случайного закрытия вкладки, обновления страницы или перехода по ссылке.
- **Проверено вживую** через Playwright: `mediaSession.playbackState` корректно проходит `playing→paused→playing→none` по стадиям старт/пауза/резюме/стоп; счётчик обращений к `wakeLock.request` увеличивается после симулированного цикла hidden→visible; `beforeunload` действительно помечается как `defaultPrevented` во время записи и корректно перестаёт быть таким после стопа.
- Коммит `b5fb993`, запушено в `origin/main`.

## Эпик 2: US-2.1/2.2/2.3 — лимит загрузки, битые файлы, язык распознавания

Перед стартом уточнили лимит файла — сейчас в коде было 2 ГБ, в спеке 1 ГБ; пользователь подтвердил привести к спеке. Дедупликацию по хэшу (нужна отдельная колонка в БД) отложили как отдельный заход, всё остальное — за один проход.

- **US-2.1**: `MAX_UPLOAD_BYTES` 2 ГБ → 1 ГБ. Каждая загрузка теперь прогоняется через `ffprobe` (новый общий `getFfprobeDurationSeconds` в `ffmpeg.js`, до этого дублировался только внутри `pipelineDiarizer.js`) — реальная длительность сохраняется в `duration_seconds` (колонка была в схеме с самого начала, но её никто никогда не заполнял), запись длиннее 4 часов (`MAX_UPLOAD_DURATION_SECONDS`) отклоняется на моменте загрузки, а не где-то в асинхронном воркере. Объединение нескольких файлов в одну встречу сознательно не трогали — это та же переделка модели хранения, что и отложенная ранее US-1.5, нет смысла делать её половинчато здесь.
- **US-2.2** (без дедупликации): тот же вызов ffprobe на загрузке заодно ловит битые/неподдерживаемые файлы — если ffprobe не смог прочитать файл вообще, это самый дешёвый надёжный сигнал «файл повреждён». Новый `backend/src/uploadValidation.js` с `UploadValidationError` — HTTP-роут загрузки и internal-колбэк recorder-бота (`completeSelfHostedMeetingRecording`) оба теперь ловят её и отвечают понятно (400 + `code`, либо помечают запись `failed` с текстом ошибки) вместо необработанного исключения.
- **US-2.3**: в уже существующий JSONB `diarization_config` пользователя добавлено поле `language` (`''` = автоопределение, `ru`/`en` — миграция не понадобилась, это просто новый ключ в существующем блобе). Ручной выбор всегда побеждает над тем, что определил ASR. ## Эпик 3 (Offline-first): планирование архитектуры + Phase A

Следующий по счёту эпик из сверки бэклога — самый крупный: полностью отсутствует (ни IndexedDB, ни service worker, ни докачки, ни статусов синхронизации). Пользователь попросил сначала спланировать архитектуру, не писать код сразу — зашли в Plan Mode.

### Планирование

Запустили Explore-агента, чтобы точно картировать текущий поток данных во `App.jsx` (нет роутера, нет query-библиотеки — `@tanstack/react-query`/`@trpc/client` установлены, но нигде не используются, всё вручную через `useState`+`apiFetch`). Запустили Plan-агента с полным контекстом для проработки решения. Ключевые архитектурные решения (все обоснованы соответствием стилю проекта — минимум зависимостей, никаких тяжёлых абстракций):

- **Не react-query** для слоя синхронизации, несмотря на то что он уже установлен — решает только половину задачи (чтение), очередь на запись с докачкой всё равно пришлось бы писать вручную.
- **Докачка по частям — свой протокол**, не tus (`tus-node-server` не знает о модели владения записями в Postgres, интеграционных издержек столько же, сколько и от самописного варианта, но плюс новая зависимость).
- **Состояние докачки — в Postgres + отдельная постоянная staging-директория**, не Redis (эфемерный, только под BullMQ) и не `tmpdir()` (может быть tmpfs, стирается при рестарте).
- Статус синхронизации — **отдельный клиентский `syncState`**, не смешивается с реальным серверным `status` (разные CSS-неймспейсы, чтобы не столкнуться).

### Найденный тестированием баг: пилюля пропадала после перезагрузки

При проверке «переживает ли неотправленная запись перезагрузку страницы» пилюля синхронизации после релоада не отображалась — но проверка напрямую через `indexedDB.open('voxmate')` подтвердила, что сама запись (Blob + `syncState: 'sync-failed'`) в очереди сохранна, это была не потеря данных, а баг отображения.

Причина — в `withQueuedRecordings` (`App.jsx`): серверная строка записи создаётся на первом шаге `syncOneItem` (`POST /api/recordings`), **до** того, как загружен хоть один байт аудио. Дедупликация в мёрдже сравнивала `serverRecordingId` очереди со списком уже известных серверу id и, как только строка появлялась на бэкенде, исключала синтетическую (несущую `syncState`) карточку в пользу голой серверной — даже если аудио ещё не докачано или докачка вовсе провалилась. В штатном потоке это не было заметно (список не перезапрашивается с сервера в процессе одной сессии), баг проявлялся именно при `loadRecordings()` после релоада/смены фильтра. Исправлено инверсией приоритета: пока элемент есть в очереди, синтетическая карточка всегда побеждает голую серверную строку, а не наоборот — как только `processQueue` реально завершает синхронизацию, элемент удаляется из очереди, и на следующей загрузке списка уже показывается настоящая серверная запись.

- **Проверено вживую** тем же одноразовым Docker-стеком + Playwright (`page.route()` для точечного перехвата чанков — надёжнее гонки с `context.setOffline()` по таймингу):
  - Базовый офлайн-цикл: запись через пикер при `context.setOffline(true)` → карточка и пилюля «✓ на устройстве» появляются мгновенно без сети → восстановление сети → автосинхронизация без участия пользователя → пилюля пропадает, реальный `status: uploaded`.

  - Перезагрузка с зависшей в очереди записью (после фикса бага выше): пилюля «⚠ ошибка синхронизации» видна сразу после релоада и остаётся видна 5 секунд подряд, при этом сама запись в IndexedDB не менялась — устойчивое поведение, не гонка.
  - `page errors: []` во всех трёх прогонах.
- Коммит `c1fe007`, запушено в `origin/main`. **На сервер (91.186.210.222) не выкатывали** — как и Epic 2, требует отдельного `git pull` + пересборки бэкенда там; фронтенд-часть тоже нужно пересобрать и задеплоить.

## Эпик 4: Обработка и статусы (автоконвейер, ретраи, отмена, concurrency)

Следующий эпик по порядку после офлайн-фёрст. Сверка: US-4.1 (понятные статусы) не начато, US-4.2 (параллельная обработка) частично (UI не блокировался, но воркер по факту был однопоточным, отмены не было вовсе), US-4.3 (автовосстановление на сбое этапа) не начато, US-4.4 (сигнал о тихом звуке) уже закрыто в эпике 1 (US-1.6) — не трогали.

Перед стартом — архитектурное планирование (Plan Mode, план в `scalable-hatching-horizon.md`, тот же файл, что и для эпика 3, — перезаписан под новую задачу). Пользователь подтвердил два решения: (1) протокол и извлечение задач становятся частью **автоматического** конвейера, а не отдельной ручной кнопки; (2) concurrency воркера = 3. Плюс два уточнения по честности: процент для «расшифровываем» — **эвристическая оценка по времени** (реального прогресса от ASR-провайдеров физически нет), отмена активной обработки — **мягкая/кооперативная** (проверка флага на границах стадий, не мгновенный AbortController через все внешние вызовы).

### Бэкенд

- Миграция `014_pipeline_stages`: `recordings.status` получил `transcribing`/`summarizing` (старое `processing` оставлено в constraint как legacy-значение, но новый код его больше не пишет — не стали форсировать backfill старых строк на проде во время деплоя), `recordings.failure_count`, `processing_jobs.cancel_requested`.
- **Ключевое архитектурное решение** — `worker.js`: `processRecording()` теперь **checkpoint-резюмируемый**: перед стартом проверяет, существует ли уже `transcripts`-строка для записи — если да, пропускает расшифровку и идёт сразу в генерацию протокола. Это делает существующий BullMQ-ретрай (тот же `attempts`/`backoff`, без второй очереди/job-типа) естественно постадийным: попытка 1 падает на протоколе → BullMQ ретраит тот же job → job видит готовый транскрипт → повторяет только протокол, не трогая расшифровку.
- Отмена: `checkCancelled()` бросает `UnrecoverableError` (BullMQ прекращает ретраи немедленно, не дожидаясь исчерпания `attempts`) на двух точках — перед стартом и на границе расшифровка→протокол. Новый роут `POST /api/recordings/:id/jobs/cancel`: если job ещё не подхвачен воркером (`waiting`/`delayed` в BullMQ) — удаляется сразу и помечается `failed`; если уже выполняется — только флаг, воркер сам остановится на ближайшей границе. Отмена не увеличивает `failure_count` (это не системная ошибка).
- `concurrency: Number(process.env.RECORDING_WORKER_CONCURRENCY || 3)` в `new Worker(...)` — раньше не задавался вообще (BullMQ-дефолт = 1, записи по факту обрабатывались строго по одной).
