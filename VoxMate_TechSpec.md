|----------|----------|
| Версия документа | 2.0 |
| Дата | 27 июня 2026 |
| Статус | Рабочий черновик (итерационная сборка) |
| Способ реализации | Claude Code, поблочно |
| Целевая среда | VPS на Ubuntu, всё в контейнерах (Docker) |

> **Что изменилось относительно v1.** Убрана вся монетизация (тарифы, оплата, гейтинг по подписке) — вынесена за рамки текущего этапа. Offline-режим переформулирован честно: запись без сети → отложенная обработка при появлении сети → система алертов. Захват аудио разведён на два подхода, бот-участник конференций вынесен в отдельный поздний этап. Документ переструктурирован под итерационную сборку: каждый блок самодостаточен, имеет критерий готовности (Definition of Done) и, по сути, является готовым рабочим заданием для Claude Code. Зафиксированы целевая инфраструктура (**VPS на Ubuntu**) и **готовность к масштабированию** как сквозное требование с самого первого блока. Вся AI-обработка (ASR + LLM) переведена на **единый шлюз OpenRouter** — один ключ, смена модели через env, без привязки к РФ-контуру. Добавлено сквозное требование **единой адаптивной дизайн-системы** (§3.8).

## 0. Принципы этого документа

1. **Поблочная сборка.** Собираем блок → проверяем по DoD → только потом следующий блок. Не начинаем блок N+1, пока блок N не прошёл свой критерий готовности.
2. **Каждый блок — рабочее задание.** Формулировки даны так, чтобы блок можно было отдать Claude Code как единицу работы с понятным «готово/не готово».
3. **Монетизация — вне рамок.** Это осознанное решение, а не упущение. Все ограничения по тарифам, оплата и подписки исключены. Когда дойдём — вернём отдельным слоем (он не ломает архитектуру ниже).
4. **Честные формулировки.** Где требование — это цель, которую надо проверить замером, так и написано: «цель, проверяется в блоке N».
5. **Целевая среда — VPS на Ubuntu.** Всё разворачивается в контейнерах (Docker + Docker Compose) на Ubuntu-VPS. Конфигурация — только через переменные окружения (12-factor), чтобы один и тот же образ ехал на любой хост без правок кода.
6. **Готовность к масштабированию — обязательна с самого начала.** Стартуем на одном VPS, но в архитектуре не должно быть решений, мешающих горизонтальному росту: приложение stateless, состояние вынесено во внешние сервисы (PostgreSQL, Redis, S3), воркеры масштабируются отдельно от веба, тяжёлая обработка идёт через очередь, ничего важного не лежит на локальном диске инстанса. Любой компонент должен выноситься на отдельный хост без переписывания кода.

## 1. Описание продукта

VoxMate AI — Progressive Web Application (PWA) для записи, транскрипции и анализа встреч. Полный цикл: голосовая запись → транскрипция → структурированный протокол с извлечением задач в формате «кто-что-когда» → отправка в интегрированные каналы (Битрикс24, Email, Telegram).

**Два подхода к захвату аудио (по нарастанию сложности):**

| Подход | Что делает | Этап |
|--------|------------|------|
| **A. Прямая запись** | (а) запись с микрофона телефона, (б) запись с микрофона компьютера через браузер | Базовый (блоки 1–2) |
| **B. Бот-участник** | Бот подключается к Zoom / Google Meet / Teams по ссылке, записывает встречу как тихий участник | **Третий этап** (блок 9) |

> **Важно по подходу B.** PWA в браузере физически не может тянуть системный звук из стороннего приложения конференции. Поэтому захват «во время звонка» решается не в клиенте, а отдельной бот-инфраструктурой, которая заходит в конференцию как участник. Это отдельный продукт по сложности — поэтому он последним.

### 1.1 Целевая аудитория (без тарифной привязки)

| Сегмент | Сценарии |
|---------|----------|
| Частные пользователи | Запись интервью, личных встреч, голосовых заметок; быстрая транскрипция через Telegram-бота |
| Менеджеры и фрилансеры | Протоколы встреч, извлечение задач, отправка в Б24/Telegram/Email |
| Команды | Группировка встреч по проектам, план-факт по задачам |

### 1.2 Платформы

| Платформа | Формат | Особенности |
|-----------|--------|-------------|
| Windows / macOS / Linux | PWA (браузер) | Запись через микрофон, установка как PWA, offline-запись |
| iOS | PWA | «Добавить на главный экран», Safari; offline-запись в IndexedDB |
| Android | PWA + TWA | Chrome, Trusted Web Activity; offline-запись |
| Telegram | Бот | Standalone: голосовое/файл → транскрипт → задачи |

## 2. Архитектура системы

Клиент-серверная архитектура. На клиенте — **offline-first запись**. На сервере — **асинхронный конвейер обработки на очереди** (это backbone, без него длинные встречи кладут систему). Всё разворачивается в контейнерах на **Ubuntu-VPS**; каждый компонент (веб, воркеры, БД, Redis, хранилище) спроектирован так, чтобы выноситься на отдельный узел и масштабироваться независимо (см. §3.7).

### 2.1 Технологический стек

**Фронтенд:**

| Компонент | Технология | Назначение |
|-----------|------------|------------|
| UI | React 19 + TypeScript 5.x | Компонентный интерфейс, типобезопасность |
| Стили | Tailwind CSS + shadcn/ui | Утилитарные стили, базовые компоненты |
| State | Zustand | Управление состоянием |
| Query | TanStack Query | Кэширование, синхронизация, статусы задач |
| Router | TanStack Router | Клиентская маршрутизация |
| PWA | Vite PWA Plugin | Service Worker, Manifest, Background Sync |
| Offline-хранилище | IndexedDB (через Dexie) | Очередь записей до отправки на сервер |

**Бэкенд:**

| Компонент | Технология | Назначение |
|-----------|------------|------------|
| Runtime | Node.js 20+ | Серверное окружение |
| HTTP | Hono | HTTP-сервер, middleware |
| API | tRPC v11 | Типобезопасный RPC |
| ORM | Drizzle ORM | БД, миграции |
| База данных | **PostgreSQL** (по умолчанию) | Основное хранилище. *Альтернатива: MySQL 8 — тогда ID через UUIDv7/ULID, а не UUIDv4* |
| Очередь | **BullMQ + Redis** | Асинхронный конвейер обработки, ретраи, статусы |
| Кэш / сессии / rate-limit | Redis | Тот же инстанс |
| Хранилище файлов | S3-совместимое (Yandex Object Storage / MinIO) | Аудио-записи |

**AI и речевые технологии — единый шлюз OpenRouter (один ключ, модель меняется конфигом):**

Вся AI-обработка идёт через **OpenRouter** — и распознавание речи, и работа LLM. Это даёт один ключ (`OPENROUTER_API_KEY`), единый биллинг и **смену модели простой строкой в env, без правок кода**. Не привязываемся к российскому контуру (осознанное решение — см. §3.3 про трансграничную передачу).

| Функция | Через | Модель (по умолчанию, меняется через env) | Примечание |
|---------|-------|--------------------------------------------|------------|
| ASR (речь → текст) | OpenRouter `/api/v1/audio/transcriptions` | `OPENROUTER_ASR_MODEL` — старт на GPT-4o Transcribe (качество) / Whisper Large V3 Turbo (дёшево/быстро) | 99+ языков вкл. русский; биллинг per-second/per-token. Файлы кодируются в base64 |
| LLM (протокол, задачи) | OpenRouter `/api/v1/chat/completions` | `OPENROUTER_LLM_MODEL` — топовый инструмент-следующий (класс Claude Sonnet / Gemini Pro) для извлечения; дешёвый (класс Gemini Flash / GPT-mini) для лёгких шагов | Конкретные слаги подобрать/проверить замером на русских встречах |
| Диаризация | по возможности из ASR-провайдера; иначе отдельный шаг | — | PyAnnote (Python+GPU) отложен |

> **Почему OpenRouter, а не прямые провайдеры.** Один интерфейс на ASR, LLM (и при желании TTS), один ключ, прозрачная смена модели через переменную окружения. Это ровно тот «меняем модель ключом», который нужен.
>
> **Но адаптер всё равно нужен.** Вызовы ASR/LLM изолируем за внутренним портом-интерфейсом, чтобы при необходимости (приватность, цена на объёме) уйти на self-hosted (Whisper/T-One на своей GPU) или прямого провайдера, не переписывая конвейер. OpenRouter — провайдер по умолчанию за этим адаптером, а не жёсткая привязка.
>
> **Длинное аудио всё равно чанкуем.** Провайдеры ограничивают размер файла (например, Whisper — до 25 МБ на файл), поэтому нарезка длинных встреч на сегменты обязательна (см. блок 2).

## 3. Сквозные требования (касаются всех блоков)

### 3.1 Система алертов (по твоему явному запросу)

Не отдельная фича, а сквозной слой, который наполняется по мере появления блоков.

**События:**

| Событие | Когда | Каналы по умолчанию |
|---------|-------|---------------------|
| `sync_done` | Оффлайн-запись долетела до сервера | in-app |
| `processing_started` | Конвейер взял встречу в работу | in-app |
| `processing_done` | Готов протокол + задачи | in-app + Web Push + (опц.) Telegram |
| `processing_failed` | Падение после исчерпания ретраев | in-app + Web Push + (опц.) Telegram/Email |
| `send_done` / `send_failed` | Результат отправки в Б24/TG/Email | in-app |

**Каналы:**
- **In-app** — бейдж статуса на встрече + центр уведомлений. Базовый, всегда.
- **Web Push** — пуш при закрытом приложении (PWA Push API). Главный сценарий: «расшифровка готова».
- **Telegram / Email** — для `done`/`failed`, опционально, по настройке пользователя.

**Принцип:** алерт всегда привязан к состоянию (`ProcessingJob.state` / `Meeting.status`), а не шлётся «вслепую». UI читает статус, пуши шлёт сервер.

### 3.2 Offline-first запись (твой фиксированный сценарий)

Нет сети → пишем локально → расшифровываем при появлении сети → алерт.

- Аудио пишется через `MediaRecorder` и складывается в **IndexedDB** как элемент очереди (с метаданными: source, длительность, локальный id).
- **Service Worker + Background Sync API** дожидается сети и выгружает запись в S3 через сервер.
- Транскрипция оффлайн **не делается** (ASR — облако/сервер). Это честное ограничение, заложенное в дизайн. On-device Whisper в браузере не используем (качество на русском низкое, телефон греется).
- После успешной выгрузки → событие `sync_done` → встреча уходит в конвейер → `processing_*`.

### 3.3 Соответствие 152-ФЗ (важно, т.к. пишем чужие голоса)

- Согласие на запись/обработку фиксируется (`User.consentRecordedAt`), для встреч с участниками — предупреждение «звонок записывается».
- Данные (аудио, транскрипты) хранятся на инфраструктуре в РФ.
- **Важный нюанс:** AI-обработка (ASR + LLM) идёт через OpenRouter, т.е. аудио и транскрипты уходят к внешним провайдерам **за пределы РФ**. Это сознательное решение (мы не ограничиваемся РФ-контуром). Следствие: трансграничную передачу нужно явно отразить в тексте согласия и политике обработки; если позже понадобится строгий РФ-контур — переключаем ASR/LLM-адаптер на self-hosted (Whisper/T-One) без переписывания конвейера.
- Право на удаление: каскадное удаление встречи + аудио в S3 + производных.
- Это требование к продукту, а не к коду одного блока — закладывается с блока 0.

### 3.4 Обработка реального («грязного») аудио

Реальные встречи — это шум, перекрытие речи, дальний микрофон, акценты. Требования:
- Поддержка длинного аудио (до 4 часов) через **чанкинг** (блок 2).
- Пользовательский словарь терминов/имён для повышения точности (можно отложить, но заложить место в модели).
- Явная обработка случая «плохое аудио → низкая уверенность ASR» с пометкой в результате.

### 3.5 Наблюдаемость и устойчивость

- **Sentry** (ошибки фронта и бэка), структурные логи, базовые метрики конвейера (длина очереди, время обработки, доля падений).
- Ретраи на каждом этапе конвейера (BullMQ), идемпотентность воркеров.
- Бэкапы БД, версионирование схемы (миграции Drizzle).

### 3.6 Аутентификация (решение зафиксировать)

- По умолчанию: **Yandex ID** + **Telegram Login** + email/пароль как fallback. OAuth 2.0 + JWT (access/refresh).
- Выбор провайдеров под RU-аудиторию и связку с Telegram-ботом. *Подтвердить набор провайдеров.*

### 3.7 Масштабируемость и развёртывание (важно — закладывается с блока 0)

**Целевая среда:** VPS на **Ubuntu**, всё в контейнерах (**Docker + Docker Compose**). Конфигурация — только через переменные окружения (12-factor): один образ запускается на любом хосте без правок кода.

**Принципы, обеспечивающие масштабируемость (соблюдаются с самого начала, не «потом»):**

- **Приложение stateless.** Сессии/JWT — в Redis, ничего критичного в памяти процесса. Любой инстанс обрабатывает любой запрос → можно поднять N копий за балансировщиком.
- **Состояние — только во внешних сервисах:** PostgreSQL, Redis, S3-совместимое хранилище. На app-боксе состояние не живёт. App-инстансы — «скот, а не питомцы».
- **Никакой зависимости от локального диска** для того, что должно пережить рестарт: всё аудио и производные — в S3, не на файловой системе инстанса.
- **Веб и воркеры — раздельные процессы.** HTTP-сервер и воркеры конвейера (BullMQ) масштабируются независимо: тяжёлую ASR/LLM-нагрузку наращиваем добавлением воркеров, не трогая веб.
- **Очередь — главный механизм масштабирования.** Приём запросов отвязан от обработки; пики не кладут систему, а растягиваются по очереди.
- **Идемпотентные воркеры.** Повтор задачи безопасен → можно гонять несколько воркеров параллельно и спокойно ретраить.
- **ASR/LLM за адаптерами.** Масштабирование — сменой провайдера или добавлением (в т.ч. GPU-) воркеров под self-hosted-движки.

**Лестница масштабирования (явный путь, без преждевременного усложнения):**

1. **Старт:** один Ubuntu-VPS, все компоненты в Docker Compose.
2. Вынести **PostgreSQL / Redis / S3** на отдельные узлы (managed или свои).
3. Поднять **несколько инстансов приложения** за балансировщиком (nginx/Caddy).
4. Выделить **отдельный пул воркеров** (включая GPU-воркеры, если перейдём на self-hosted ASR).
5. **Оркестрация (Kubernetes) — только когда реально упрёмся.** До этого момента её не вводим.

**Рычаги по БД на будущее:** пул соединений (PgBouncer), read-реплики, партиционирование тяжёлых таблиц (`meeting`, транскрипты) по дате/пользователю. Закладывать сразу не нужно, но схема не должна этому мешать.

> Принцип короткой формулой: **«стартуем на одном VPS, но пишем так, будто завтра разъезжаемся на много машин»**. Ни одно решение в коде не должно предполагать, что сервер ровно один.

### 3.8 Дизайн-система и адаптивность (важно — закладывается с блока 0)

Весь продукт — в **единой современной дизайн-системе**, которая красиво и одинаково работает на телефоне, планшете и десктопе. Один источник правды, не разрозненные стили по экранам.

**Один источник правды:**
- **Слой токенов** (Tailwind config + CSS-переменные) — единственное место, где заданы цвета, типографика, отступы, радиусы, тени, анимация. Компоненты ходят за значениями только сюда.
- **Один компонентный слой** — shadcn/ui (на Radix), кастомизированный под токены. Используется везде: PWA, web-страницы бота, лендинг. Никаких параллельных наборов кнопок/инпутов.
- **Light + dark** обе темы — переключаются на уровне токенов.

**Токены (что задаём):** семантические цвета (surface / elevated / text / muted / accent / success / warning / danger + состояния hover/active/disabled); типографическая шкала (display / heading / body / caption / mono); шкала отступов; радиусы; уровни теней (elevation); длительности и easing для анимаций.

**Направление (предложение — нужно утвердить).** Субъект продукта — «порядок из голоса»: превращаем шумную живую речь в чистую структуру (протокол, задачи). Отсюда язык дизайна: спокойный, сфокусированный, с точной типографикой и моноширинным шрифтом для таймкодов/данных. Стартовый набор:
- **Шрифты:** Geist Sans (интерфейс) + **Geist Mono** (таймкоды, длительности, статусы) — современная, чистая пара. *Если хочешь, чтобы VoxMate был в одной семье с твоими продуктами — у тебя уже есть узнаваемый стиль (тёмная тема + Geist + лаймовый акцент #B1E258), его можно расширить на этот продукт.*
- **Акцент:** один яркий акцентный цвет на действия и состояние «готово/слушает». Конкретный hex — за тобой (твой лайм — готовый вариант).
- Не уходим в шаблонные AI-дефолты (кремовый фон + serif + терракота; либо чёрный фон с единственным кислотным акцентом ради эффекта). Берём направление под субъект, а не «как у всех».

**Адаптивность («красиво на всех устройствах»):**
- **Mobile-first**, текучие лейауты, заданные брейкпоинты; проверка на реальных телефоне/планшете/десктопе.
- **Тач-таргеты ≥ 44px**, удобство одной рукой на телефоне.
- **Safe-area** (вырезы/нотчи) на iOS; PWA при установке выглядит нативно (иконки, splash, theme-color).

**Quality floor (соблюдается с блока 0, не «потом»):** адаптив до мобайла, видимый keyboard-focus, уважение `prefers-reduced-motion`, контраст не ниже WCAG AA.

**Тексты интерфейса — часть дизайн-системы** (и заодно совпадает с твоим стилем без воды): активный залог, sentence case, действие называется одинаково через весь флоу («Отправить» → тост «Отправлено»). Пустые экраны и ошибки говорят, **что делать**, а не извиняются и не льют воду. Называем вещи так, как их видит пользователь («уведомления», а не «webhook config»).

**Где живёт:** дизайн-система (токены + базовые компоненты) поднимается в **блоке 0**, все последующие блоки наследуют её. Это гарантирует единый вид с первого экрана.

### Блок 1 — Запись + offline-хранение + sync + Алерты v1

### Блок 2 — Транскрипция: очередь + ASR + чанкинг + Алерты v2

**Цель:** превратить выгруженное аудио в транскрипт через асинхронный конвейер.
**Объём:**
- BullMQ + Redis: очередь обработки, воркер. **Веб и воркеры — раздельные процессы; число воркеров наращивается независимо от веб-сервера.**
- Этап `chunk`: нарезка длинного аудио (до 4 ч) на сегменты — обязательна, т.к. провайдеры ограничивают размер файла (напр. Whisper — до 25 МБ).
- Этап `asr`: интеграция ASR через **адаптер** поверх OpenRouter `/api/v1/audio/transcriptions` (модель из `OPENROUTER_ASR_MODEL`). Сборка транскрипта из сегментов с сохранением таймкодов.

- `ProcessingJob` с состояниями; ретраи; идемпотентность.
- Алерты v2: `processing_started` / `processing_done` / `processing_failed` + Web Push на `done`.

**DoD:**
- [ ] Загрузил 2-часовое аудио → получил полный транскрипт.
- [ ] При искусственном падении ASR воркер ретраит, при исчерпании — статус `failed` + алерт.
- [ ] Web Push «расшифровка готова» приходит при закрытом приложении.
- [ ] Замерено фактическое время обработки относительно длины аудио (проверка цели «< 0.5×»).

**Claude Code:** конвейер, очередь, адаптер — да. Подбор параметров чанкинга и стыковки сегментов — итеративная настройка на реальных файлах.

### Блок 3 — Протокол + задачи «кто-что-когда» (ядро продукта)

**Цель:** из транскрипта получить читаемый протокол и надёжно извлечённые задачи. **Это ядро ценности — ему отдельный блок и метрики качества.**
**Объём:**
- Этап `llm` через адаптер поверх OpenRouter `/api/v1/chat/completions` (модель из `OPENROUTER_LLM_MODEL`): транскрипт → структурированный протокол (повестка, решения, риски) + задачи.
- **Стратегия длинных транскриптов:** не полагаемся на то, что 4-часовая встреча влезет в один промпт. Делаем map-reduce (поэтапная свёртка сегментов → итоговый протокол) либо используем длинноконтекстную модель — выбор за моделью в env, но код не должен предполагать единый влезающий промпт.
- Извлечение задач в формате «кто-что-когда»: исполнитель, действие, срок.
- **Разрешение относительных дат** («к пятнице», «через неделю» → конкретная дата от даты встречи).
- Базовое сопоставление исполнителя по имени (полное сопоставление с контактами — блок 4).
- **Набор для оценки качества**: размеченный датасет из N реальных встреч, метрики precision/recall по задачам. Цель по качеству задаётся и проверяется здесь.

**DoD:**
- [ ] На тестовом наборе встреч задачи извлекаются с согласованной целевой метрикой (например, recall ≥ X, precision ≥ Y — цифры зафиксировать).
- [ ] Относительные даты корректно превращаются в абсолютные.
- [ ] Протокол человекочитаем и структурирован.
- [ ] Промпты и few-shot вынесены в версионируемые файлы (не захардкожены).

**Claude Code:** скаффолд пайплайна и парсинг — да. **Качество извлечения — итеративная работа с промптами и проверка на реальных данных, не one-shot.** Это самый «человеческий» блок.

| **Offline** | Запись без сети | полная (запись + локальное хранение) |
| | Обработка | только при наличии сети (осознанное ограничение) |
| | Синхронизация | Background Sync API + fallback |
| **Безопасность / 152-ФЗ** | Аутентификация | OAuth 2.0 + JWT |
| | Хранение аудио | шифрование at rest (AES-256), данные в РФ |
| | Передача | TLS 1.3 |
| | Удаление данных | каскадное, по запросу пользователя |
| **Среда / развёртывание** | Хостинг | **VPS на Ubuntu**, Docker + Docker Compose |
| | Конфигурация | только через переменные окружения (12-factor) |
| **Масштабируемость** *(базовое требование)* | Старт | один Ubuntu-VPS, все компоненты в контейнерах |
| | Готовность к росту | **обязательна с блока 0:** stateless-приложение, состояние во внешних сервисах (PostgreSQL/Redis/S3), веб и воркеры масштабируются независимо, обработка через очередь |
| | Путь масштабирования | один VPS → вынос БД/Redis/S3 → несколько инстансов за балансировщиком → отдельный пул воркеров (вкл. GPU) → Kubernetes только при необходимости |
| **Совместимость** | Браузеры | Chrome 90+, Firefox 88+, Safari 14+, Edge 90+ |
| | ОС | Windows 10+, macOS 12+, iOS 14+, Android 8+ |

> Цифры, помеченные «цель», не утверждаются как гарантии до замера в соответствующем блоке.

- **Монетизация целиком:** тарифы, оплата, рекуррентные подписки, гейтинг по подписке, лимиты по тарифам, SLA-уровни. Вернём отдельным слоем — архитектура выше его не блокирует.

- **On-premise / корпоративные кастомные интеграции.**
- **Self-hosted ASR/LLM** (T-One/Whisper, GigaChat on-prem) — переключаемся через адаптер, когда/если потребуется по объёму или приватности.
