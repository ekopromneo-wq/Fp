ПРОБЛЕМЫ ПРИ РАБОТЕ С СЕРВИСОМ SPEECH2TEXT (speech2text.ru)
============================================================
Собрано из SESSION_LOG.md и кода (backend/src/speech2textDiarizer.js,
meetingBot.js, worker.js). Дата сводки: 2026-07-18.

Speech2Text использовался в двух ролях:
  а) как один из методов диаризации готовой аудиозаписи (speech2textDiarizer.js);
  б) как сторонний «бот-участник» конференций, которого отправляют на встречу
     по ссылке (meetingBot.js).


1. BITRIX24 С КАСТОМНЫМ ДОМЕНОМ НЕ ПОДДЕРЖИВАЕТСЯ (бот-участник)
--------------------------------------------------------------
Кастомный домен портала (bitrix.ekoprom.org) Speech2Text не распознаёт как
валидный URL конференции — ошибка «meeting_url is not a supported conference
URL». Принимаются только домены bitrix24.ru / bitrix24.com.
→ Источник: SESSION_LOG 2026-07-05, §3. Один из триггеров решения строить
  собственный self-hosted recorder-bot вместо стороннего бота Speech2Text.


2. [СНЯТО 2026-07-20] «НЕТ API ДЛЯ ПРОВЕРКИ БАЛАНСА» — ПУНКТ БЫЛ ОШИБОЧНЫМ
-------------------------------------------------------------------------
Изначально здесь значилось, что метода баланса у сервиса нет и состояние счёта
приходится определять косвенным зондом. Это неверно: в PDF-документации API
описан GET /api/user/amounts, отдающий баланс числом:
  {"minutes":{"available":300,"used":0},"amount":0}
  minutes.available / minutes.used — минуты по тарифу;
  amount — рубли на лицевом счёте (с него доплачиваются задачи сверх тарифа).
Проверено вживую 2026-07-20 нашим ключом: HTTP 200, 300 минут доступно.
Неверный ключ и пустой ключ → HTTP 401 с JSON {"name":"Unauthorized",...}.
Рядом есть GET /api/recognitions/TASK-ID/price (расчёт стоимости задачи:
{"duration":1.9,"pricePerMinute":0.4,"price":0.76}) и PATCH
/api/recognitions/TASK-ID/pay (списание с лицевого счёта).
Причина ошибки — документацию прочли не полностью, метод искали по имени
/api/balance. Исправлено: balances.js теперь показывает реальные минуты.
→ Источник: PDF «API Speech2Text — Документация», раздел «Просмотр доступных
  лимитов минут и оплата»; живая проверка 2026-07-20.


3. НЕТ ОПРЕДЕЛЕНИЯ ИМЁН СПИКЕРОВ — ТОЛЬКО ОБЕЗЛИЧЕННЫЕ МЕТКИ
----------------------------------------------------------
Speech2Text отдаёт только generic-метки (SPEAKER_N / «Спикер N»), без попытки
определить, кто говорит (в отличие от Shopot и Gemini). API отдаёт голосовые
векторы спикеров, но они не используются. Для сценариев, где нужны имена,
метод не подходит.
→ Источник: SESSION_LOG 2026-07-05, §1; комментарий в speech2textDiarizer.js
  (строки 146–151), parseSpeech2TextResult подставляет «Спикер N».


4. API-КЛЮЧ ПЕРЕДАЁТСЯ В URL КАК QUERY-ПАРАМЕТР
----------------------------------------------
Ключ уходит в строке запроса (?api-key=<KEY>) на всех эндпоинтах: submit,
статус, результат. Это риск утечки ключа в логи/прокси/историю, в отличие от
передачи в заголовке.
→ Источник: speech2textDiarizer.js (submit — стр. 43, poll — стр. 65,
  result — стр. 89, checkStatus — стр. 106).


5. URL ВСТРЕЧИ ДЛЯ БОТА СНАЧАЛА НЕ ВАЛИДИРОВАЛСЯ ПО ДОМЕНУ (дыра)
---------------------------------------------------------------
Путь createMeetingBotRecording принимал произвольный meetingUrl и передавал
его в Speech2Text без проверки домена (в отличие от собственного recorder-bot,
где allowlist уже был). Готовый isSupportedMeetingUrl / SUPPORTED_PLATFORM_HOSTS
существовал в meetingBot.js, но не вызывался. Исправлено — подключили проверку;
теперь чужой URL (напр. evil.example.com) отбивается 400 до обращения к сервису.
→ Источник: SESSION_LOG (аудит безопасности), пункт 4; проверено вживую.


6. АСИНХРОННАЯ МОДЕЛЬ С ОПРОСОМ И НЕСТАНДАРТНЫМИ КОДАМИ СТАТУСА
-------------------------------------------------------------
Распознавание асинхронное: submit задачи → опрос статуса → отдельный запрос
результата. Коды статуса нестандартные и зашиты по числам:
  200 — готово, 204 — речь не обнаружена (возвращаем null), 501 — ошибка
  распознавания. Интервал опроса 5 c, таймаут 30 мин (по истечении — падение
  с ошибкой timeout). Любой сбой на любом шаге → метод возвращает null и вся
  цепочка откатывается на запасной путь (OpenRouter ASR + текстовая нарезка).
→ Источник: speech2textDiarizer.js (pollSpeech2TextJob стр. 61–86,
  transcribeWithSpeech2Text стр. 153–172).


7. РУЧНОЙ ВЫБОР ЯЗЫКА (US-2.3) НЕ ПРИМЕНЯЕТСЯ К SPEECH2TEXT
---------------------------------------------------------
Пользовательская настройка языка транскрипта не пробрасывается в Speech2Text —
сервис работает по своей языковой логике (параметр lang, по умолчанию 'ru' из
SPEECH2TEXT_LANGUAGE). Осознанно не трогали, честно отметили в примечании под
настройкой. Итоговое значение transcripts.language — это метаданные и может не
совпадать с тем, что реально учитывал сервис.
→ Источник: SESSION_LOG 2026-07 (US-2.3), стр. лога про diarization_config.language.


8. СЛЕДСТВИЕ: ОТКАЗ ОТ СТОРОННЕГО БОТА В ПОЛЬЗУ СОБСТВЕННОГО РЕКОРДЕРА
--------------------------------------------------------------------
Из-за (1) непокрытия Bitrix24 и требования пользователя «захватывать звонки без
третьих сервисов» решено строить собственный self-hosted recorder-bot
(Node + Playwright + PulseAudio/ffmpeg). Speech2Text как бот-участник остался
лишь запасным движком для части платформ; Yandex Telemost при этом на самом
Speech2Text-боте вживую отрабатывал (включая транскрипт).
→ Источник: SESSION_LOG 2026-07-05, §3–§5.


9. НЕПОСЛЕДОВАТЕЛЬНЫЙ СИНТАКСИС АВТОРИЗАЦИИ (разные схемы в одном API)
--------------------------------------------------------------------
Один и тот же API-ключ передаётся ПО-РАЗНОМУ в разных семействах эндпоинтов:
  • Файловое распознавание (/api/recognitions/...) — ключ как QUERY-ПАРАМЕТР
    в URL: ?api-key=<KEY>   (speech2textDiarizer.js, balances.js).
  • Бот-участник встреч (/api/meetings/join, /api/meetings/<id>/stop) — тот же
    ключ как ЗАГОЛОВОК: Authorization: Bearer <KEY>   (meetingBot.js).
Это непоследовательность в синтаксисе самого API: интеграция вынуждена держать
два разных способа аутентификации к одному сервису. Как побочный эффект — часть
запросов таскает ключ в URL (см. пункт 4), часть в заголовке.
→ Источник: сверка speech2textDiarizer.js (стр. 43/65/89/106, `?api-key=`)
  против meetingBot.js (стр. 34/62, `Authorization: Bearer`).


10. КОДЫ СТАТУСА ПЕРЕИСПОЛЬЗОВАНЫ КАК ПРИКЛАДНЫЕ ВНУТРИ JSON
-----------------------------------------------------------
Числа HTTP-статусов используются не как HTTP-код ответа, а как прикладной
статус ЗАДАЧИ внутри тела JSON — поле status.code:
  200 = распознавание готово, 204 = речь не обнаружена, 501 = ошибка.
То есть HTTP-ответ приходит 200 OK, а реальное состояние надо доставать из
body.status.code и сверять с этими числами вручную. Нестандартно и легко
перепутать с настоящими HTTP-статусами.
→ Источник: pollSpeech2TextJob (speech2textDiarizer.js, стр. 72–80).


11. ЗАДАЧА НЕ ПАДАЕТ ПРИ НЕХВАТКЕ МИНУТ, А ВСТАЁТ В ПАУЗУ (status 102)
--------------------------------------------------------------------
Если тарифных минут не хватает на длительность аудио, распознавание не
завершается ошибкой — задача переходит в status.code = 102 / value = "paused"
(«Приостановлено: лимит доступных минут исчерпан») и ждёт, пока её доплатят
через /pay или пополнят тариф. Сама она не возобновится.
Наш файловый поллер про 102 не знал и трактовал его как «ещё не готово»:
крутил опрос все 30 минут таймаута и только потом уходил в откат. Исправлено
2026-07-20 — теперь падаем сразу с внятным сообщением про исчерпанные минуты.
Поллер бота-участника (worker.js) раньше паузу игнорировал (оставлял запись в
processing) — из-за этого paused-встреча висела бы бесконечно. Исправлено
2026-07-20: value === 'paused' теперь закрывает запись как failed с внятным
сообщением и пишет в журнал подключений событие 'paused' (US-15.1). Компромисс:
если минуты позже пополнят и сервер возобновит задачу, мы её уже не подхватим
(поллер смотрит только status = processing) — сознательно выбрали «явный отказ»
вместо «вечного processing».
→ Источник: PDF, таблица статусов (queued 30 / paused 102 / done 200 / done 204
  / error 501); speech2textDiarizer.js pollSpeech2TextJob; worker.js
  pollMeetingBotRecordings + failMeeting; botLog.js (событие 'paused').


12. ЕЩЁ ЕСТЬ WEBHOOK — НО ОН НЕ ЗАМЕНЯЕТ НАШ ОПРОС ТАК ПРОСТО, КАК КАЗАЛОСЬ
--------------------------------------------------------------------------
В API есть /api/webhook/create|get|update|delete: при смене статуса
транскрипции сервис сам шлёт POST на наш URL (событие
transcription.status_changed, поле secret_token = наш API-ключ, ретраи
5/10/15/20 минут, затем failed). На первый взгляд — можно выкинуть опрос.
Разобрались (2026-07-20): выигрыш небольшой, а стоимость выше ожидаемой.
  • Файловый путь (speech2textDiarizer, опрос 5 c) webhook НЕ упрощает: это
    один синхронный воркер-джоб submit→poll→result; webhook там только заставил
    бы джоб блокироваться в ожидании входящего вызова — хуже, чем сейчас.
  • Остаётся только опрос бота-участника (worker.js, 30 c). Но приёмник webhook
    живёт в HTTP-процессе (index.js/recordings.js), а логика ingest
    (ingestCompletedMeeting/failMeeting) — в ОТДЕЛЬНОМ процессе-воркере. Значит
    нужен либо вынос ingest в общий модуль, либо дублирование через очередь.
  • Приёмник — публичный эндпоинт; входящий вызов от speech2text локально не
    протестировать (нужен реальный PUBLIC_APP_URL на проде). Норма проекта —
    «проверено вживую», а этот путь отсюда не проверяется.
Итог: выигрыш ~30 c латентности на уже работающем пути против кросс-процессного
рефактора + непроверяемого публичного эндпоинта. Решено ОТЛОЖИТЬ до момента,
когда (а) появится прод-стенд для живой проверки входящего вызова и (б) ingest
всё равно будет выноситься в общий модуль по другой причине. Опрос оставлен как
рабочий и проверенный механизм.
→ Источник: PDF, разделы «Webhook API» / «Входящий webhook (payload)»;
  index.js (serve) vs worker.js (poll) — раздельные процессы.
